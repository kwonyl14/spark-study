# 1. Optimizing Executors

- **Resource Planning**
    
    : yarn & HDFS 클러스터에서 실행 중인 spark job 을 튜닝하고자 할 때, 
    
    - 노드 4개, 각 노드당 64GB RAM, 각 노드당 16코어 CPU
- **Fat & Skinny Executors 문제**
    - **Fat executors** : 많은 리소스를 가진 적은 executor
    - **Skinny executor** : 적은 리소스를 가진 많은 executor
    - **Fit executors** : 딱 적절한 executor

1. **Fat executors**
    
    ```scala
    spark.executor.memory = 64g
    spark.executor.cores = 16
    spark.executor.instances = 4
    
    // 실행 명령
    spark-submit --num-executors 4 --executor-memory 64g --executor-cores 16
    ```
    
    - **장점**
        - 매우 큰 task 처리 가능
        - executor 내부 병렬성 극대화 가능
    - **단점**
        - HDFS 및 동시 I/O 불리
        - 64GB 는 컨테이너 메모리에 맞지 않을 가능성 있음
        - exeutor 하나 죽으면 → 클러스터 25% 다운

1. **Skinny executors** 
    
    ```scala
    spark.executor.memory = 4g
    spark.executor.cores = 1
    spark.executor.instances = 64
    
    // 실행 명령
    spark-submit --num-executors 64 --executor-memory 4g --executor-cores 1
    ```
    
    - **장점**
        - 좋은 I/O 처리량
        - 많은 작은 task 에 유리할 수도 있음
    - **단점**
        - 각 executor 는 싱글 스레드
        - 큰 task 는 OOM 발생 가능
        - executor 가 너무 많아 관리 오버헤드 증가

1. **Fit executors**
    1. **시스템용 리소스 확보**
        - 각 머신당 4GB RAM, 4코어 → yarn, OS, HDFS 용
        - 남은 자원 : 60GB RAM, 12코어 * 4대
    2. **ApplicationMaster 고려**
        - 약 1GB 를 할당 ← 큰 클러스터에서는 영향이 미미
    3. **Executor 코어 수 결정**
        - HDFS 처리량을 위해 executor당 ≤ 5코어 유지
        - 48코어 / 5 ≈ 9 executors
    4. **Executor 메모리 계산**
        
        ```scala
        (4 × 60 – 1) / 9 = 26GB
        ```
        
        → 오버헤드 7-8% 확보 ⇒ 실제 executor memory ≈ 24GB
        
    5. **실행**
        
        ```scala
        spark-submit --num-executors 9 --executor-memory 24g --executor-cores 5
        ```
        
    - **단점**
        - 4대 머신에 9 executors 는 완벽하게 균등하지는 않다.

1. **Dynamic Resource Allocation (동적 리소스 할당)**
    
    : Spark 가 실행 중에 executor 를 요청하거나 종료해주는 기능
    
    - **언제 사용?**
        - 멀티 테넌시, 클러스터 공유 환경, 클러스터가 유휴 상태일 때, 장기 실행 job
        
        → 저지연 작업에서는 성능에 영향을 줄 수도 있다. 
        
    - **동작 방식**
        - 일정 시간 이상 task 가 대기하면 → 새 executor 요청
        - 라운드 로빈 방식으로 executor 증가
        - task 큐가 계속 남아있으면 → 지수적으로 증가
    - **주요 설정**
        
        ```scala
        // 동적 리소스 할당 기능을 켜거나 끄는 스위치
        spark.dynamicAllocation.enabled
        // 애플리케이션 시작 시 처음 요청할 executor 수
        spark.dynamicAllocation.initialExecutors
        // Spark가 절대 줄이지 않는 최소 executor 수
        spark.dynamicAllocation.minExecutors
        // Spark가 최대 몇 개까지 executor를 늘릴 수 있는지 제한
        spark.dynamicAllocation.maxExecutors
        // Task가 대기 상태로 이 시간 이상 있으면 executor를 새로 요청
        spark.dynamicAllocation.schedulerBacklogTimeout
        // 다음 executor 요청까지 기다리는 시간
        spark.dynamicAllocation.sustainedSchedulerBacklogTimeout
        // Executor가 이 시간 동안 아무 일도 안 하면 제거
        spark.dynamicAllocation.executorIdleTimeout
        
        **// *** 단 항상 ! minExecutors ≤ executor 수 ≤ maxExecutors**
        ```
        
        - 앱 시작 → initialExecutors 요청
        - 작업 몰림 → schedulerBacklogTimeout 이후 확장 시작
        - backlog 지속 → sustainedSchedulerBacklogTimeout 마다 지수 증가
        - 작업 줄어듦 → executorIdleTimeout 지나면 executor 제거

# 2. Data Skews

- **데이터 스큐**
    
    : 데이터가 균일하게 분포되어 있지 않으면 태스크도 균일하게 분배되지 않는다.
    
    **→ 발생하는 문제 : Straggling Tasks (느린 태스크)**
    
    - 특정 태스크가 유난히 오래 걸림
    - 해당 Stage 완료가 지연됨
    - 전체 job 완료 시간이 늘어남
    
    → 해당 문제는 리소스를 늘려도 소용이 없다. 병목은 CPU 부족이 아니라, 특정 파티션에 데이터가 몰린 구조적 문제이기 때문이다.
    
    ```scala
    package part5boost
    
    import common._
    import org.apache.spark.sql.{Dataset, SparkSession}
    import org.apache.spark.sql.functions._
    
    object FixingDataSkews {
      val spark = SparkSession.builder()
        .appName("Fixing Data Skews")
        .master("local[*]")
        .getOrCreate()
    
      // broadcast join 비활성화
      // 기본적으로 Spark는 작은 테이블을 broadcast 해서 shuffle을 피하려고 함
      // 하지만 우리는 의도적으로 shuffle join을 발생시켜
      // 데이터 스큐 문제를 재현하려는 것이므로 broadcast를 끔
      spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)
    
      // Dataset 변환을 위한 implicit import
      import spark.implicits._
    
      // 기타(Guitar) 데이터 생성
      // 40000개의 랜덤 기타 객체 생성 후 Dataset으로 변환
      // make, model 값이 랜덤 생성되므로 특정 key에 데이터가 몰릴 가능성 존재
      val guitars: Dataset[Guitar] =
        Seq.fill(40000)(DataGenerator.randomGuitar()).toDS
    
      // 기타 판매 데이터 생성
      // 200000개의 랜덤 판매 기록 생성
      // 실제로는 판매 데이터가 훨씬 많으므로 join 시 skew 발생 가능성 높음
      val guitarSales: Dataset[GuitarSale] =
        Seq.fill(200000)(DataGenerator.randomGuitarSale()).toDS
    
      /*
        문제 정의
    
        Guitar와 GuitarSale이 유사하다고 판단하는 조건:
    
        1) make와 model이 동일해야 함
        2) soundScore 차이가 0.1 이하
    
        목표:
        각 Guitar(configurationId별)에 대해
        "유사한 GuitarSale들의 평균 판매가"를 계산하라.
    
        예시:
        Gibson L-00, soundScore 4.3 이라면
        soundScore가 4.2 ~ 4.4 범위인 모든 Gibson L-00 판매 기록의 평균 가격을 구함
       */
    
      def naiveSolution() = {
    
        // 1. Join 수행
        // make, model 기준으로 inner join
        // 여기서 shuffle 발생 가능성 매우 높음
        // 특정 make, model 값이 많으면 하나의 파티션에 몰릴 수 있음 (data skew)
        val joined = guitars
          .join(guitarSales, Seq("make", "model"))
    
          // 2. Range 조건 필터
          // soundScore 차이가 0.1 이하인 경우만 필터링
          // equality join이 아니라 조건 join이므로
          // Spark가 더 많은 데이터를 처리하게 됨
          // join 이후 필터 → 이미 많은 데이터가 모인 상태일 수 있음
          .where(
            abs(guitarSales("soundScore") - guitars("soundScore")) <= 0.1
          )
    
          // 3. groupBy 수행
          // configurationId 기준 집계
          // groupBy는 wide transformation → 또 한 번 shuffle 발생
          // 이미 skew된 상태에서 다시 shuffle이 일어나면
          // 특정 task가 과도하게 오래 걸릴 수 있음 (straggler task)
          .groupBy("configurationId")
    
          // 평균 판매가 계산
          .agg(
            avg("salePrice").as("averagePrice")
          )
    
        // 실행 계획 출력
        // explain()을 통해 physical plan 확인 가능
        // 여기서 Exchange가 보이면 shuffle 발생
        joined.explain()
    
        // 실제 action 발생
        // count()가 호출되어야 Spark가 lazy evaluation을 끝내고
        // 실제 job을 실행함
        joined.count()
      }
    
      def main(args: Array[String]): Unit = {
    
        // naive 방식 실행
        // 이 방식은 skew가 발생할 가능성이 매우 높음
        naiveSolution()
        Thread.sleep(1000000)
      }
    }
    ```
    

1. **Salting 기법**
    
    : spark 에서 데이터 스큐를 해결하기 위한 가장 대표적인 방법으로, 데이터를 인위적으로 더 잘게 쪼개서 균등 분산시키는 방법이다.
    
    - salt 값을 추가하면 데이터는 n+1 개의 키로 분산된다. 그 중 하나는 균일한 분포를 가진 salt 키
        - salt 구간이 클수록 → skew 가능성은 줄지만, shuffle 크기는 커진다.

1. **Salting 동작 방식**
    - 작은 RDD 또는 DF 에 대해서 각 행마다 여러 개의 salt 값을 붙여서 explode 시킨다.
    - 다른 RDD/DF 에는 각 행마다 salt 구간에서 랜덤 값을 하나 추가한다.
    - 조인 시에는 기존 key+salt 컬럼을 합쳐서 join 한다.
    - 필요하다면 부분 결과들을 다시 합쳐준다.
    
    ** salting 은 join 뿐만 아니라, groupBy/aagregation 같은 연산에도 적용이 가능하다.
    

1. **실습**
    
    ```scala
      /*
        Skew 해결 버전 (Salting 적용)
        기존: 특정 make, model 값이 많으면 하나의 파티션으로 몰림
        해결: (make, model, salt) 즉, 하나의 key를 100개의 작은 key로 분산시킴
       */
      def noSkewSolution() = {
    
        // ----------------------------------------------------------
        // 1. 작은 쪽 Dataset을 explode 하여 salt 값 0~99 추가
        // ----------------------------------------------------------
        // 각 guitar row가 100배로 증가
        // (0~99까지 salt 값을 붙여서 모든 경우를 미리 만들어둠)
        //
        // 예:
        // Gibson L-00 → Gibson L-00 + salt=0
        // Gibson L-00 → Gibson L-00 + salt=1
        // ...
        // Gibson L-00 → Gibson L-00 + salt=99
        //
        // 이렇게 하면 join key가 100배로 쪼개짐
        val explodedGuitars =
          guitars.withColumn(
            "salt",
            explode(lit((0 to 99).toArray))
          )
    
        // ----------------------------------------------------------
        // 2. 큰 쪽 Dataset에는 랜덤 salt 값 부여
        // ----------------------------------------------------------
        // monotonically_increasing_id()는 증가하는 ID 생성
        // % 100을 통해 0~99 사이 값 생성
        //
        // 결과:
        // 같은 make, model이라도
        // 서로 다른 salt 값을 가지게 되어 분산됨
        val saltedGuitarSales =
          guitarSales.withColumn(
            "salt",
            monotonically_increasing_id() % 100
          )
    
        // ----------------------------------------------------------
        // 3. Salt 포함 join 수행
        // ----------------------------------------------------------
        //
        // 기존: join(make, model)
        // 변경: join(make, model, salt)
        //
        // → 특정 key가 100개로 분산되므로
        // → 하나의 executor가 몰빵 처리하는 현상 방지
        val nonSkewedJoin =
          explodedGuitars
            .join(
              saltedGuitarSales,
              Seq("make", "model", "salt")
            )
    
            // soundScore 범위 조건 유지
            // salting은 분산만 바꾼 것이지
            // 비즈니스 로직은 동일
            .where(
              abs(
                saltedGuitarSales("soundScore") -
                explodedGuitars("soundScore")
              ) <= 0.1
            )
    
            // groupBy는 여전히 shuffle 발생
            // 하지만 이미 데이터가 균등하게 분산된 상태이므로
            // 특정 task만 느려지는 현상 방지
            .groupBy("configurationId")
    
            // 평균 판매가 계산
            .agg(
              avg("salePrice").as("averagePrice")
            )
    
        // 실행 계획 확인
        // Exchange가 여러 번 보일 것
        // 하지만 partition 크기가 균등해짐
        nonSkewedJoin.explain()
    
        // action 발생
        // 실제 실행
        nonSkewedJoin.count()
      }
    ```

# 3. Serialization Problems

1. **직렬화(Serialization)**
    
    : 객체를 전송 가능한 형태로 바꾸는 과정이다. 메모리에 살아있는 객체를 바이트 덩어리로 변환하는 것이다.
    
2. **왜 직렬화가 필요한지**
    - **Driver JVM** : 코드 작성 및 실행 계획 수립
    - **Executor** **JVM** : 실제 계산 수행
    
    **RDD 나 DS** 에서 람다 함수를 사용하면 Driver 에서 만들어지고, Executor JVM 으로 전송되어 거기에서 실제로 실행이 된다. 즉, **람다 함수는 네트워크를 타고 이동한다. 이때 필요한 게 바로 직렬화(Serialization)다.**
    
    ```scala
    rdd.map(x => x * 2)
    ```
    
    - **Applicable to lambdas (RDDs and Datasets only)**
        
        **DF/SQL** 에서 직렬화 문제가 잘 발생하지 않는 이유는 **DF/SQL** 은 Expression Tree 기반의 람다가 아니라 **표현식을 사용**하기 때문에 → **람다 통째로 X
        - Spark 내부 Expression → Catalyst 가 해석 → Spark 가 코드 생성**
        
        ```scala
        rdd.map(x => x * 2)
        ```
        
    
3. **Task serialization needed**
    
    : Task 직렬화가 필요하다.
    
    - Driver 는 Executor 에서 실행하기 위해 람다를 전송해야 한다.
    - 람다가 참조하는 모든 객체도 함께 직렬화되어야 한다.
    - 만약 참조 대상이 클래스/객체의 필드라면, 그 클래스도 직렬화 대상이 된다. 하지만 모든 객체가 직렬화가 가능한 것은 아니다.

1. **직렬화 문제가 발생하는 이유**
    
    ```scala
    class MyService {
      val connection = new DatabaseConnection()  // 직렬화 불가능
    
      def process(rdd: RDD[Int]) = {
        rdd.map(x => {
          connection.save(x)  // 여기서 터짐
        })
      }
    }
    ```
    
    람다 함수에서 connection 을 참조할 때, connection 은 직렬화가 불가능한데, Driver 가 Executor 로 람다를 네트워크를 통해 전달할 때 → Spark 는 이 객체는 보내지 못한다고 판단하여 → NotSerializableException
    
    ⇒ 람다 내부에서 참조한 모든 객체가 함께 직렬화되어야 하기 때문이다.
    
    - **발생하는 연산 유형**
        - RDD.map, RDD.filter, Dataset.map, Dataset.flatMap …
        
        → 기타 JVM 함수 기반 Transformation
        
        → 이런 JVM 람다 기반 연산에서 람다가 외부 객체를 참조하면 발생한다.
        

1. **직렬화 문제 해결 방법**
    - **make classes/objects Serializable (클래스/객체를 Serializable로 만들기)**
        
        ```scala
        // Java 의 마커 인터페이스 Serializable 를 사용함
        class MoreGeneralRDDMultiplier **extends Serializable** {
          val factor = 2
        
          def multiplyRDD() =
            rdd.map(_ * factor).collect().toList
        }
        // 클래스 필드를 람다가 쓰면, 클래스 전체가 직렬화 대상
        ```
        
        - 사용하기 쉽지만, 오버헤드가 크다.
        - 클래스 전체가 직렬화 대상이 되기 때문에 필요없는 필드까지 네트워크로 전송될 수 있다.
    - **capture local values (로컬 값 캡처하기)**
        
        ```scala
        // 전체 객체 직렬화를 피하고 최소 값만 보내기
        class MoreGeneralRDDMultiplierEnclosure {
          val factor = 2
        
          def multiplyRDD() = {
            val enclosedFactor = factor
            rdd.map(_ * enclosedFactor).collect().toList
          }
        }
        // 람다는 enclosedFactor 만 캡처 → Int 값 하나만 직렬화 → 객체 전체는 안 보냄
        ```
        
        - 필요한 값만 직렬화해서 클래스 전체가 캡처되지 않도록 한다.
        - 코드를 신경써서 작성해줘야해서 초보자에게는 어려울 것 같다.
    - **중첩 객체 구조**
        
        ```scala
        // Nested object 내부에서 외부 클래스의 factor 사용
        class MoreGeneralRDDMultiplierNestedClass {
          val factor = 2
        
          object NestedMultiplier extends Serializable {
            val extraTerm = 10
            val localFactor = factor
        
            def multiplyRDD() =
              rdd.map(_ * localFactor + extraTerm).collect().toList
          }
        }
        // Nested 객체를 Serializable, 외부 factor를 localFactor 로 캡처
        ```
        
        - 중첩 구조에서는 어디에서 무엇을 캡처하는지가 가장 중요하다.
    - **생성자 인자 캡처**
        
        ```scala
        // 생성자 파라미터를 사용하는 경우
        class LegalDrinkingAgeChecker(legalAge: Int) {
          def processPeople() = {
            val ageThreshold = legalAge // 로컬 변수로 복사
            people.map(_.age >= ageThreshold).collect().toList
          }
        }
        // legalAge는 클래스 필드처럼 동작 → 바로 쓰면 객체 전체 직렬화
        ```
        
        - 생성자 인자도 필드처럼 취급되기 때문에 반드시 로컬 값으로 캡처해야 한다.

1. **Serializable 없이 구조적으로 해결하는 방법**
    - **함수 객체 만들기**
        
        ```scala
        class DrinkingAgeChecker(legalAge: Int) {
        
          val check = {
            val capturedLegalAge = legalAge
            age: Int => age >= capturedLegalAge
          }
        }
        ```
        
        - method 가 아니라 Function1 객체를 생성한다. FunctionX 타입은 기본적으로 Serializable 여서, val 로 만들어야 한다.
    - **람다를 인자로 전달**
        
        ```scala
        class DrinkingAgeFlagger(checker: Int => Boolean) {
          def flag() = {
            val capturedChecker = checker
            people.map(p => capturedChecker(p.age)).collect().toList
          }
        }
        ```
        
        - 람다를 로컬 값으로 캡처한다. (최소 단위만 직렬화)

# 4. Kryo Serialization

- **Kryo** : 빠르고 메모리 효율적인 직렬화 방식으로, Spark 기본 직렬화는 Java Serialization 인데, Kryo 는 그것보다 훨씬 빠르고 메모리도 덜 쓴다.
하지만 모든 객체는 자동으로 직렬화할 수 없고, 클래스 등록이 반드시 필요하다.
- **직렬화를 할 때** : shuffle, cache, executor 간 데이터 이동
- **Kryo 설정 방법**
    
    ```scala
    val sparkConf = new SparkConf()
    
    sparkConf
      **.set("spark.serializer", "org.apache.spark.serializer.KryoSerializer")
      .set("spark.kryo.registrationRequired", "true")**
      **.registerKryoClasses(
        Array(
          classOf[Person],
          classOf[Array[Person]]
        )
      )**
    ```
    
    - Kryo를 serializer로 설정 → Spark 기본 직렬화를 Kryo로 교체
    - 클래스 등록 필수 설정 → 반드시 사용할 클래스를 등록해주어야 함
    - 직렬화할 클래스 등록 → Person 클래스, Person 배열 등록
- **SparkSession 에 설정 적용**
    
    ```scala
    val spark = SparkSession.builder()
      .appName("Kryo serialization")
      **.config(sparkConf)**
      .master("local[*]")
      .getOrCreate()
    ```
    
    - 만든 설정을 SparkSession 에 전달
- **Kryo 와 Java 직렬화 비교**
    
    
    | **항목** | **Java** | **Kryo** |
    | --- | --- | --- |
    | 속도 | 느림 | 매우 빠름 |
    | 메모리 사용 | 많음 | 적음 |
    | 설정 필요 | 없음 | 클래스 등록 필요 |
    | 안정성 | 높음 | 제한적 |
- **Kryo 언제 사용..**
    - 대규모 셔플이 많을 경우
    - 캐시를 많이 사용하는 경우
    - 사용자 정의 객체(RDD, Dataset)을 사용하는 경우
    - 메모리 압박이 심한 경우
