1. Core

  1. Spark Layered Architecture

    • Storage 계층
      • files, HDFS, S3, databases
    • Cluster manager
      • standalone(bundled with spark)
      • YARN
      • Mesos
      • Kubernetes
    • Spark Core
      • RDD 는 Spark 의 가장 기본 단위 (first citizens) → Spark 초창기부터 변하지 않음
      • DataFrame & SQL 은 많은 자동 최적화 기능 제공
    • High-level libs
      • Streaming, ML, GraphX → 모든 라이브러리는 Spark Core(RDD) 를 기반으로 동작

  2. Spark Architecture

    Spark 에서는 worker 노드들이 실제 작업을 수행하는 역할을 한다.

    • Executor
      • worker 의 논리적 노드 (JVM)
      • 하나의 애플리케이션을 위해 작업 수행
      • 한 애플리케이션에서 여러 개 존재 가능
      • 독립적인 메모리/CPU 자원을 가진 JVM 컨테이너
    • Driver
      • Spark 애플리케이션의 메인 JVM
      • 애플리케이션 하나 당 Driver 하나
      • 작업을 시작하면 executor 에게 작업을 분배하는 역할
    • Cluster manager
      • executor 배포 관리
      • Driver 가 executor 와 자원을 요청하는 역할

    성능을 위해서 Driver 는 worker 와 물리적으로 가까이 있어야 좋고, worker 들끼리도 가까워야 shuffle 비용이 감소한다. (물리적 거리 = 성능 차이)

  3. RDD / DataFrame / Dataset

  • RDD : 분산된 JVM 객체들의 타입이 있는 컬렉션
    • 특징 : Spark 의 가장 기본 단위로, 상위 API 들도 결국 RDD 로 환원됨
    • 장점 : 분산 제어 가능, 순서 제어 가능, SQL 로 표현 불가능한 복잡한 연산 가능
    • 단점 : 사용하기 어려움, Spark 내부 동작을 알아야 고급 연산 가능, 빠른 데이터 처리를 위한 API 가 부족함
  • DataFrame : 고수준 분산 데이터 구조
    • 특징 : Row 기반, 스키마 존재, SQL 지원, SQL 최적화 후 내부적으로 RDD 생성
    • 장점 : 사용 쉬움, SQL 지원, Spark 가 자동 최적화
    • 단점 : 타입 안전하지 않음, 모든 계산 표현 불가, 추가 최적화 어려움
  • Dataset[T] : 타입이 있는 분산 컬렉션
    • 특징 : DataFrame 의 SQL 기능 지원, RDD 의 함수형 연산 지원 (DataFrame = Dataset[Row])
    • 장점 : 타입 안전, SQL + 함수형 프로그래밍 모두 가능, 일부 자동 최적화 지원
    • 단점 : JVM 객체 생성 비용 큼, 람다 최적화 불가능

    • RDD 는 특수한 연산이 필요할 때만 사용하는 것이 좋다.

    • Spark 실행 원리

      • Lazy Evaluation (지연 실행) : Spark 는 마지막 순간까지 실행하지 않는다.

    • Planning (계획 수립)

      1. DF/SQL → RDD 변환
      2. RDD 변환들을 그래프로 구성
      3. Logical Plan 생성
      4. Physical Plan 생성
      5. 최적화 수행

      Spark 는 컴파일러가 동작한다.

    • Transformations vs Actions

      • Transformations : 새로운 RDD/DF 반환 → 실행되지 않음
      • Actions : 실제 실행 트리거 → 결과 반환 (Action 이 호출되면 Job 시작)

    • Spark App Executor

      • Spark 애플리케이션 실행
        • Action → Job 발생
      • Job → Stage 로 분할
        • Stage 는 이전 Stage 완료 후 시작
        • Stage 수는 최소화하는 것이 좋음
      • Stage → Task 로 분할
        • Task = 가장 작은 작업 단위
        • 하나의 Partition 을 처리
        • executor 에서 실행
    개념관계
    1 Job1개 이상의 Stage
    1 Stage1개 이상의 Task
    1 Task1 Partition 처리
    1 Executor여러 Task 실행 가능
    1 Partition하나의 Executor에 존재

    // RDD[Array[String]]

    • Narrow transformation : 파티션끼리 서로 알 필요가 없기 때문에, 셔플 없음
    val empDetails = empTokens.map(tokens => (tokens(4), tokens(7)))

    step 3 : 필요한 정보만 뽑아서 튜플로 만드는데, 여기서도 셔플 없음

    • 5번째 컬럼 (token(4)), 8번째 컬럼 (token(7)) 두 개로 튜플 생성 RDD[(String, String)] ← (token(4)) : 부서 / (token(7)) : 급여의 경우 (부서, 급여) ← 모든 행을 사용하지만, 해당 컬럼만 뽑아오는 경우
    val empGroups = empDetails.groupByKey(2)

    step 4 : 데이터를 부서별로 그룹핑한다.

    • Wide transformation : 모든 파티션이 고려되어야 하기 때문에, 셔플 있음

      (Shuffle : 데이터가 executor 들 사이에서 이동 → 매우 비쌈..)

    • repartition(n) : 파티션 개수만 조정 (데이터를 다시 골고루 섞음)

    • groupByKey(n) : 키 기준으로 데이터를 모으면서 결과 파티션을 n개로 나눔

    val avgSalaries = empGroups.mapValues(salaries => 
    	salaries.map(_.toInt).sum / salaries.size)

    step 5 : 급여 문자열 → int 변환 후 집계한 값을 개수로 나눠서 평균을 계산해준다.

    • RDD[(부서, 평균급여)]

    앞 전체 코드 흐름을 Stage1/Stage2 로 구분해서 보여준다.

    → 중간에 groupByKey 때문에 shuffle 이 생겨서 stage 가 분리된다.

  2. Task / Stage / Shuffle

    • Task : 가장 작은 계산 단위로, 하나의 파티션에 대해 한 번 실행되며, 하나의 executor 에서 실행된다.
    • Stage : Task 들을 포함하는 단위로, 데이터 교환이 없음을 강제한다. (=파티션이 서로 데이터를 필요로 하지 않는 상태), 이전 Stage 에 의존한다. (=이전 Stage가 끝나야 시작 가능)
    • Job / Stage / Task

      • Spark Application = SparkContext(또는 SparkSession) 실행 단위

        # Application1
        rdd1.count()              // Job 1
        rdd1.map(_*2).count()      // Job 2
        rdd1.repartition(23).count() // Job 3

      • Job = action 실행 단위

      • Stage = Shuffle 기준으로 Job 내부가 나뉜 단위

      • Task = Stage 를 파티션 단위로 쪼갠 실행 단위

    • Shuffle
      • executor 들 사이에서 데이터 교환이 발생한다.
      • Stage 사이에서 발생한다.
      • 다음 Stage 가 시작되기 전에 반드시 완료되어야 한다.

  3. 의존성 (Dependencies)

  • Narrow dependencies
    • 하나의 입력(부모) 파티션이 하나의 출력(자식) 파티션에만 영향을 주고, 계산이 빠르다.
    • ex: map, flatMap, filter, projections 등
  • Wide dependencies
    • 하나의 입력 파티션이 둘 이상의 출력 파티션에 영향을 주고, shuffle 을 포함한다. (=executor 사이 데이터 전송) 그래서 계산 비용이 크다.
    • ex: grouping, joining, sorting 등
  1. 셔플 (Shuffle)
    • 셔플이 비싼 이유
      • 데이터 전송, 직렬화/역직렬화, 셔플 파일에서 새 데이터를 로딩해야 하기 때문
    • 셔플이 성능 병목이 되는 이유
      • 데이터 교환 자체가 시간이 걸린다.
      • 다음 계산이 시작되기 전에 반드시 완전히 끝나야 한다.

3. 쿼리 실행 계획 (Query Plans)

  1. SQL Job 이 실행되는 방식

    Spark SQL은 Catalyst Optimizer 를 사용한다. SQL을 실행하면 내부적으로 다음 단계가 진행된다.

    • Unresolved Logical Plan
      • 아직 컬럼/테이블 참조가 확정되지 않은 논리적 계획
    • Resolved Logical Plan
      • 컬럼 타입, 참조 테이블 확정
    • Optimized Logical Plan
      • 불필요한 연산 제거, 필터 푸시다운, 컬럼 프루닝 등 최적화 적용
    • Physical Plan 생성
      • 실제 클러스터에서 실행할 물리적 연산 결정

  2. 쿼리 실행 계획 읽는 방법

    val ds1 = spark.range(1, 10000000)
    val ds2 = spark.range(1, 100000000, 2)
    val ds3 = ds1.repartition(7)
    val ds4 = ds2.repartition(9)
    val ds5 = ds3.selectExpr("id * 3 as id")
    val joined = ds5.join(ds4, "id")
    val sum = joined.selectExpr("sum(id)")
    sum.explain()

    sum.explain() 결과로 물리 실행 계획이 출력된다. (실행 X)

    AdaptiveSparkPlan // AQE 가 활성화되어 있음
      └─ HashAggregate (final sum)
          └─ Exchange SinglePartition
              └─ HashAggregate (partial sum)
                  └─ Project
                      └─ SortMergeJoin
                          ├─ Left branch
                          └─ Right branch
    • Query plan 의 기본 구성 요소

      • Range

        Range (1, 10000000, step=1)
        • spark.range() 는 SparkSession 기반으로 내부 데이터를 생성하는 논리적 데이터 소스이며, 실제 파일을 읽지 않고 Range Scan 단계로 실행 계획에 나타난다. 초기 파티션 수는 Spark SQL 의 설정 및 내부 계산 로직에 의해 결정된다.
        • spark.range() 의 초기 파티션 수
          • spark.sql.shuffle.partitions (기본 200)
          • 또는 Spark 가 내부적으로 잡는 partition 수

      • Exchange

        Exchange **hashpartitioning**(...)
        Exchange **RoundRobinPartitioning**(...)
        Exchange **SinglePartition**
        // -> 물리적 계획에서 이 세개 보이는 경우, shuffle 발생
        • exchange 는 해당 지점에서 shuffle 이 발생했음을 의미한다. exchange 가 수행되면 executor 간 데이터 이동이 발생하며, 이 과정에서 네트워크/디스크 I/O + 직렬화 비용이 추가로 발생한다. 또한 shuffle 을 기준으로 Stage 경계가 생성되어 이전 연산과 이후 연산이 서로 다른 Stage 로 분리된다. exchange 이후에는 파티션이 다시 구성되므로, 다음 Stage 의 Task 수와 병렬성이 새롭게 결정된다. 따라서 실행 계획에서 exchange → shuffle 발생 → Stage 분리 → 병렬 구조 재구성

      • Sort

        Sort [id ASC NULLS FIRST]
        • sort 는 데이터가 특정 컬럼 기준으로 정렬되는 것을 의미한다. 위 같은 경우, id 컬럼을 오름차순으로 정렬하며, NULL 값은 앞쪽에 배치한다. sortmergejoin 이 선택된 경우, join 수행 전에 반드시 정렬이 필요하다.
          • 각 파티션 내부에서 조인 키 기준으로 정렬
          • 정렬된 두 데이터셋을 병합 방식으로 조인 수행
        • 정렬은 CPU 사용량이 높은 연산이며, 데이터 양이 많을수록 비용이 크게 증가한다. 따라서 실행 계획에서 sort 가 보이면 성능에 영향을 줄 수 있다.

      • SortMergeJoin

        SortMergeJoin [id], [id], Inner
        • sortmergejoin 은 두 데이터셋을 정렬 기반으로 병합하여 조인하는 방식이다. 위 같은 경우, id 컬럼을 기준으로 내부 조인을 수행하는 것이다. 이 방식에서 shuffle 과 join 이 필요합니다.
          • 같은 조인 키를 가진 데이터가 반드시 같은 파티션에 있어야 한다. 이를 위해 조인 키 기준으로 hash partitioning 이 수행되며, 이 과정에서 shuffle 이 발생한다. → 조인 키 기준 shuffle(exchange)
          • 각 파티션 내부 데이터가 조인 키 기준으로 정렬되어 있어야 한다. 그래야 정렬된 두 데이터를 병합 방식으로 효율적으로 비교할 수 있다. → 파티션 내부 sort
        • 대용량 데이터 조인에서 기본적으로 사용되는 전략이며, 메모리 부담이 비교적 낮고 안정적인 대신 shuffle 과 sort 비용이 발생하는 것이 특징

      • HashAggregate

        partial_sum
        sum
        • spark 실행 계획에서 집계가 수행되고 있음을 의미한다. 실행 계획에서 위와 같이 두 번 등장하는 이유는 spark 가 집계를 두 단계로 나누어서 처리하기 때문이다. 먼저 각 파티션 내부에서 부분 집계를 수행하여 데이터를 미리 줄인 뒤, 그 결과를 shuffle 로 모아 최종 집계를 수행한다. 이와 같은 구조를 Two-phase aggrgation(2단계 집계) 라고 하며, 원본 데이터를 그대로 shuffle 하는 대신 부분 집계 결과만 이동시키므로 네트워크 비용을 줄이기 위한 spark 의 기본 최적화 전략이다.

  3. Join 전에 수행되는 준비 작업

    Query Plan 을 보면

    • repartition(7)
    • repartition(9)

    그 이후 join 직전에 아래가 수행이 된다.

    Exchange hashpartitioning(id, 200)
    Sort

    즉 Spark 는 다음 순서로 준비한다.

    1. Join을 위해 동일한 hash partitioning 수행
    2. 각 partition 내부 정렬
    3. SortMergeJoin 실행

    중요한 점은 repartition(7), repartition(9)는 어차피 200 partition 으로 다시 나뉘어서 중복 Shuffle 을 유발하기 때문에 실무에서는 이런 불필요한 repartition 은 제거하는 것이 좋다.

  4. sum(id)의 실행 과정

    집계는 항상 다음 구조를 가진다.

    1. 각 partition에서 partial sum 수행
    2. Exchange SinglePartition
    3. 최종 sum 수행

    즉 집계는 항상 부분 집계 → Shuffle → 최종 집계

    SinglePartition은 모든 결과를 하나의 partition으로 모으기 때문에 병렬성이 1로 줄어든다. 하지만 global aggregation 특성상 불가피하다.

  5. 중요한 사항

    • 몇개의 Stage 가 생기는지 (Exchange가 많을수록 Stage도 많아진다.)
      • Exchange 개수 = Shuffle 개수
      • Shuffle 개수 = Stage 경계 수
    • 몇개의 Task 가 생기는지 (Partition 수가 병렬성을 결정한다.)
      • Task 수 = Partition 수

  6. Query plan 의 특징

    Query Plan 은 실행 전에 모든 연산을 보여준다. ( ← dataFrame.explain() 으로 확인 가능)

    • partitioning 정보 포함, Stage 구조 예측 가능, Shuffle 위치 파악 가능, Task 수 예측 가능

    • explain(true) 사용 시

      Parsed Logical Plan, Analyzed Logical Plan, Optimized Logical Plan, Physical Plan 위 사항들을 출력하면서 Spark 는 진짜 컴파일러처럼 동작한다.

4. The Spark UI and DAGs

  1. DAG (Directed Acyclic Graph)

    방향이 있는 비순환 그래프로, spark 에서 DAG 은 Job 의 연산 의존 관계를 시각화한 그래프로 사용된다.

  2. Spark UI

    • Jobs : 실행된 작업 목록, 성공/실패 여부, 소요 시간, 포함된 Stage 목록

    • Stages : Task 개수, Shuffle read/write, 실행 시간, GC 시간, 실패 Task

      → 실제 성능 병목을 확인할 수 있는 곳

    • Environment : Spark 어떤 설정으로 실행되었는지 확인하는 곳

    • Executors : 메모리 사용량, GC 시간, Task 수행 수, 실패 횟수

      → OOM, GC 과다 문제를 확인할 수 있는 곳

    • Storage : cache/persist 된 RDD/DF 목록, 메모리 사용량, 디스크 사용량

      → 캐시가 제대로 되었는지 확인할 수 있는 곳

    • SQL (Spark SQL 을 사용한 경우) : Query plan 시각화, 각 Stage 구조, Physical plan

      → Query 튜닝 시 확인할 곳

  3. Spark UI 에서 확인할 부분

    • Stages : (Duration 이 가장 긴 작업에서의 병목 후보)
      • Duration 이 가장 긴 Stage
      • Shuffle read/write 작업이 가장 큰 Stage → 네트워크 I/O (join, sort, groupBy)
      • Spill 이 발생한 Stage 확인 → 메모리 부족, 대용량 aggregation/join, shuffle partition
    • Executors
      • GC time 이 과도한지 → JVM 힙 압박, 객체 과다 생성, 캐시 과다 사용 가능성
      • 특정 executor 만 느린건지 → 일부 task 만 느리다면 data skew / 전체가 느리다면 shuffle, 파티션 문제
      • 실패한 task 가 있는지
    • SQL : 물리적 계획, exchange 위치, join 타입

  4. spark UI 에서 기억할 것

    • Exchange가 보이면 Shuffle이다.
    • Shuffle이 많으면 Stage가 많아진다.
    • Partition 수가 Task 수를 결정한다.
    • Task Duration 분포를 보면 Skew를 알 수 있다.
    • Query Plan은 실행 설계도, UI는 실행 결과다.
    • 두 개를 매칭해야 정확한 병목 분석이 가능하다.

5. A Tale of Two Spark APIs

  1. RDD

    // RDD
    val rdd = sc.parallelize(1 to 1000000000)
    rdd.count()
    • 개념
      • Spark 의 가장 기본 API, 데이터 = JVM 객체, 연산 = 사용자 람다 함수, 실행 = 함수 체이닝
    • 내부 동작 방식
      • 각 파티션에서 사용자 함수 호출, 객체를 하나씩 처리, 함수 호출 오버헤드 발생, JVM 객체 생성/해제 반복, GC 부담 증가
    • 특징
      • 스키마가 없고, SQL 최적화 없음, 코드 생성 없음, Catalyst Optimizer 없음, 객체 기반 처리, 메모리 효율 낮춤

  2. DataFrame

    // DataFrame
    import spark.implicits._
    val df = rdd.toDF("id")
    df.count()
    df.selectExpr("count(*)").show()
    • 개념
      • RDD 위에 만들어진 관계형 API, 데이터 = Row + Schema, 연산 = 컬럼 기반 표현, 실행 = 쿼리 계획 → 최적화 → 코드 생성
    • 내부 표현 방식
      • 객체 대신 UnsafeRow (바이너리 포맷) 사용, 컬럼을 바이트 배열처럼 다룸, 메모리 사용량 감소, CPU 캐시 효율 증가
    • 특징
      • 스키마 존재, Catalyst Optimizer 사용, predicate pushdown 가능, projection pruning 가능, join 전략 자동 선택, WholeStageCodegen 지원, GC 부담 감소

  3. Dataset

    // Dataset
    val ds = spark.range(1, 1000000000)
    ds.count()
    ds.selectExpr("count(*)").show()
    • 개념
      • DataFrame + 타입 안정성, 내부 실행 엔진은 DF 와 동일, 추가로 Encoder 사용
    • 내부 동작 방식
      • 내부는 UnsafeRow 기반, 실행은 DF 와 동일, 객체와 Row 변환 시 Encoder 사용
    • 특징
      • DF 와 동일한 최적화, 타입 안정성 제공, 약간의 인코딩/디코딩 비용 존재

6. Deploying & Configuring Spark Applications

  1. Spark Configuration

    : Spark 애플리케이션의 동작 방식, 리소스 사용량, 병렬성, 셔플 방식 등을 결정하는 설정값 집합이다.

    • 메모리를 얼마나 쓸지, Executor 를 몇 개 띄울지, Shuffle partition 을 몇 개로 할지, 직렬화 방식을 무엇으로 할지, AQE 를 켤지 말지 등을 결정하게 된다.

  2. Spark Application 실행 (spark-submit)

    Spark application 은 spark-submit 으로 실행한다.

    • 주요 옵션

      • -executor-memory

        → 각 executor에 할당할 메모리 설정으로, executor당 RAM을 얼마나 줄 것인지 지정

      • -driver-memory

        → driver에 할당할 메모리 설정

      • -jars

        → 추가적인 JVM 라이브러리 포함

      • -packages

        → Maven 좌표로 외부 라이브러리 추가

      • -conf (key) (value)

        → 기타 Spark 설정 전달 (JVM 옵션 포함)

      • -help

        → 모든 옵션 보기

    // spark-submit
    spark-submit \
      --conf spark.executor.memory=4g \
      --conf spark.sql.shuffle.partitions=200 \
      app.jar
      
    // SparkSession
    SparkSession.builder()
      .config("spark.executor.memory", "4g")
      .config("spark.sql.shuffle.partitions", "200")
      .getOrCreate()
      
    // 실행 중 설정 변경
    spark.conf.set("spark.sql.shuffle.partitions", "100")
    • 주요 옵션
      • -class : main class 지정
      • -master : Spark cluster 주소
      • -deploy-mode : 실행 모드 (client / cluster)
      • -verbose : 상세 로그 출력
      • -supervise : 실패 시 재시작

  3. 클러스터 구조

    • Cluster Manager (Standalone, YARN, Mesos, Kubernetes)
      • 클러스터 상태를 관리하는 노드
      • 다른 노드들은 실제 작업 수행
      • driver / worker 프로세스와 통신
    • Spark Executors
      • driver가 할당한 task 실행
      • 결과와 상태를 driver에 보고
    • Spark Driver
      • stage / task 상태 관리
      • cluster manager와 통신
      • 전체 애플리케이션 제어

  4. 실행 모드 (Execution Modes)

    Spark 의 실행 모드에는 local, client, cluster 세 가지가 있다.

    • Client Mode
      • Driver 가 job 을 제출한 클라이언트 머신에서 실행되고, 클라이언트가 Spark 프로세스를 관리
      • executor에 더 많은 리소스 할당 가능, 클라이언트 노드가 죽어도 애플리케이션은 유지 가능, 결과를 즉시 로컬에서 확인 가능
      • driver 리소스 부족 가능, driver ↔ executor 통신 오버헤드, 상대적으로 느릴 수 있음
    • Cluster Mode
      • Driver가 클러스터 worker 노드에서 실행되고, cluster manager가 driver 관리
      • driver에 더 많은 메모리 할당 가능, driver와 executor 통신 빠름, 전체적으로 더 빠른 성능
      • driver가 있는 노드가 죽으면 애플리케이션 실패, executor에 할당할 리소스가 줄어들 수 있음
    • Local Mode
      • 전체 애플리케이션이 하나의 머신에서 실행되며, 개발/테스트용

  5. Spark 애플리케이션 설정 방법

    • 코드에서 설정

      SparkSession.builder()
        .config("spark.executor.memory", "1g")

    • 실행 후 spark.conf.set()

      spark.conf.set("spark.executor.memory", "1g") // 일부 설정은 실행 후 변경 불가함

    • spark-submit 에서 설정 (권장)

      spark-submit \
      --executor-memory 4g \
      --driver-memory 2g \
      --conf spark.sql.shuffle.partitions=500

  6. 배포 흐름

    코드 작성 → JAR 패키징 → spark-submit 실행 → Driver 생성 → Executors 생성 → Task 분배 및 실행

  7. 실습

    package part2foundations

    // Spark에서 데이터를 읽고 쓰기 위한 핵심 클래스들 import
    import org.apache.spark.sql.{SaveMode, SparkSession}

    object TestDeployApp {

      /*
        프로그램 실행 방법:

        spark-submit ... TestDeployApp inputFile outputFile

        args(0) = 입력 파일 경로
        args(1) = 출력 파일 경로
      */
      def main(args: Array[String]): Unit = {

        // 프로그램 실행 시 인자가 2개가 아니면 종료
        if (args.length != 2) {
          println("Need input file and output file")
          System.exit(1)
        }

        /*
          SparkSession 생성
          - Spark 애플리케이션의 진입점
          - SparkContext + SQLContext + HiveContext를 포함하는 상위 개념

          method 1:
          .config()를 이용해서 애플리케이션 시작 시 설정 지정
          여기서는 executor 메모리를 1g로 설정
        */
        val spark = SparkSession.builder()
          .appName("Test Deploy App")  // Spark UI에 표시될 애플리케이션 이름
          .config("spark.executor.memory", "1g") // Executor 메모리 설정
          .getOrCreate()

        // DataFrame DSL 사용을 위한 implicits import
        import spark.implicits._

        /*
          입력 파일을 JSON 형식으로 읽기

          option("inferSchema", "true")
          - JSON 필드 타입을 자동으로 추론
          - 별도로 schema를 지정하지 않아도 됨

          args(0)은 입력 파일 경로
        */
        val moviesDF = spark.read
          .option("inferSchema", "true")
          .json(args(0))

        /*
          DataFrame 변환 로직

          1. select:
             - 필요한 컬럼만 선택
             - 컬럼 이름 변경(as)

          2. where:
             - Major_Genre가 Comedy이고
             - IMDB_Rating이 6.5보다 큰 영화만 필터링

          3. orderBy:
             - Rating 기준 내림차순 정렬
             - null 값은 마지막으로 보냄
        */
        val goodComediesDF = moviesDF.select(
          $"Title",
          $"IMDB_Rating".as("Rating"),
          $"Release_Date".as("Release")
        )
          .where(($"Major_Genre" === "Comedy") and ($"IMDB_Rating" > 6.5))
          .orderBy($"Rating".desc_nulls_last)

        /*
          method 2:
          spark.conf.set()을 이용한 설정 변경

          ⚠ 주의:
          - 모든 설정이 런타임 중 변경 가능한 것은 아님
          - executor.memory 같은 설정은 이미 SparkContext가 시작된 이후라
            실제 executor 메모리에 반영되지 않을 수 있음
        */
        spark.conf.set("spark.executor.memory", "1g") // 경고: 일부 설정은 변경 불가

        /*
          method 3:
          spark-submit 실행 시 --conf 옵션으로 설정 전달

          예:
            spark-submit \
              --conf spark.executor.memory=1g \
              --master local[*] \
              yourJar.jar input output

          또는 전용 옵션 사용 가능:
            --master              → spark.master
            --executor-memory     → spark.executor.memory
            --driver-memory       → spark.driver.memory

          -> 실무에서는 보통 spark-submit 단계에서 설정을 전달
        */

        // 변환된 결과를 콘솔에 출력 (Action 발생)
        goodComediesDF.show()

        /*
          결과를 JSON 파일로 저장

          mode(SaveMode.Overwrite):
          - 기존 파일이 있으면 덮어쓰기

          format("json"):
          - JSON 형식으로 저장

          save(args(1)):
          - 출력 경로
          - Spark는 디렉토리 단위로 저장
            (여러 partition 파일 생성 가능)
        */
        goodComediesDF.write
          .mode(SaveMode.Overwrite)
          .format("json")
          .save(args(1))

        // SparkSession 종료 (명시적으로 호출해도 되고 안 해도 되지만 권장)
        spark.stop()
      }
    }

7. Catalyst (Catalyst Query Optimizer)

  1. Catalyst

    : Spark SQL, DF, DS 를 위한 쿼리 최적화 엔진으로, RDD 에는 없고, SQL 구조를 가진 API 에서만 동작

  2. Spark Job 내부 동작

    Spark는 SQL/DataFrame 연산이 실행되면 아래 과정을 거친다.

    • Unresolved Logical Plan (미해결 논리 계획)
      • Spark 는 연산 구조를 트리 형태로 만든다.
      • 아직 컬럼 타입이나 실제 테이블 위치는 모르고, 단순히 무엇을 하라는 구조만 있다.
    • Resolved Logical Plan (해결된 논리 계획)
      • Catalog 를 사용해서 컬럼이 어디서 오는지 데이터 타입이 무엇인지 모두 확정한다.
    • Optimized Logical Plan (최적화된 논리 계획)
      • Catalyst 가 규칙 기반 최적화를 수행한다. 트리를 분석하고 변형한다.
        • 불필요한 연산 제거, 계산 미리 수행, 조건 아래로 밀어넣기
    • Physical Plan (물리 실행 계획)
      • 실제 어떤 알고리즘을 쓸지 결정하고, 실행 코드를 생성한다.
        • HashAggregate, SortMergeJoin, BroadcastJoin, FileScan

  3. Catalyst 의 주요 최적화 기법

    • Constant Folding (상수 미리 계산)

      : 실행 전에 계산 가능한 것은 다 계산한다.

    • Filter Pushdown

      : 가능한 한 빨리 데이터를 줄인다. → shuffle 비용을 줄이기 위함

    • Rule Combining : 여러 최적화 규칙을 조합해서 더 단순한 실행 계획으로 만든다. → 불필요한 Project 제거

    • Column Pruning : 불필요한 컬럼을 읽지 않는다. → parquet 에서 엄청 중요

    • 최종 결과

  4. Logical Plan vs Physical Plan

    • Logical Plan

      : 무엇을 할지 정의, 어떻게 할지는 모름, 알고리즘 선택 없음

    • Physical Plan

      : 실제 실행 방법, Join 알고리즘 선택, Aggregate 방식 선택, Shuffle 여부 결정

  5. Catalyst 의 강점와 약점

    • 강점 : 표현이 간결하고 대부분의 계산 표현이 가능하다. 자동 최적화를 지원하기 때문에 속도 향상 !
    • 약점 : 람다 최적화 불가

  6. 실습

    package part2foundations

    import org.apache.spark.sql.{DataFrame, SparkSession}

    object CatalystDemo {

      // SparkSession 생성
      // local 모드에서 실행 (클러스터 아님)
      val spark = SparkSession.builder()
        .appName("Catalyst Demo")
        .master("local")
        .getOrCreate()

      // DataFrame DSL 사용을 위한 implicits import
      import spark.implicits._

      // JSON 파일을 읽어 DataFrame 생성
      // inferSchema = true → 컬럼 타입 자동 추론
      val flights = spark.read
        .option("inferSchema", "true")
        .json("src/main/resources/data/flights")

      // ------------------------------------------------------------
      // 1. 여러 개의 where 체이닝
      // Catalyst는 이런 chained filter들을 하나의 Filter로 합쳐버린다.
      // ------------------------------------------------------------

      val notFromHere = flights
        .where($"origin" =!= "LGA")
        .where($"origin" =!= "ORD")
        .where($"origin" =!= "SFO")
        .where($"origin" =!= "DEN")
        .where($"origin" =!= "BOS")
        .where($"origin" =!= "EWR")

      // explain(true)
      // 1. Parsed Logical Plan
      // 2. Analyzed Logical Plan
      // 3. Optimized Logical Plan
      // 4. Physical Plan
      //
      // Optimized Logical Plan에서
      // 여러 개의 where가 하나의 Filter 조건으로 합쳐지는 것을 확인할 수 있다.
      notFromHere.explain(true)

      // ------------------------------------------------------------
      // 2. 팀별로 따로 만든 필터 (중복 & 비효율적 코드 상황 가정)
      // Catalyst가 이를 자동 정리해준다.
      // ------------------------------------------------------------

      // 팀1: origin != LGA && dest == DEN
      def filterTeam1(flights: DataFrame) =
        flights
          .where($"origin" =!= "LGA")
          .where($"dest" === "DEN")

      // 팀2: origin != EWR && dest == DEN
      def filterTeam2(flights: DataFrame) =
        flights
          .where($"origin" =!= "EWR")
          .where($"dest" === "DEN")

      // 두 팀 필터를 합쳐서 적용
      val filterBoth = filterTeam1(filterTeam2(flights))

      // Catalyst는 여기서:
      // 1. 동일한 조건 중복 제거
      // 2. 필터 결합
      // 3. 가능한 경우 하나의 Filter 노드로 압축
      filterBoth.explain(true)

      // ------------------------------------------------------------
      // 3. Filter Pushdown 예시 (데이터 소스 단계까지 필터 밀어넣기)
      // ------------------------------------------------------------

      // 먼저 JSON 데이터를 Parquet으로 저장
      // Parquet은 컬럼 기반 포맷 → pushdown 가능
      flights.write.save("src/main/resources/data/flights_parquet")

      // Parquet 파일을 읽으면서 필터 적용
      val notFromLGA = spark.read
        .load("src/main/resources/data/flights_parquet")
        .where($"origin" =!= "LGA")

      // explain
      //
      // 여기서 중요한 부분:
      // PushedFilters: [Not(EqualTo(origin,LGA))]
      //
      // 이 의미는
      // Spark가 파일을 읽은 후 필터하는 것이 아니라
      // 파일 읽기 단계에서 이미 조건을 적용한다는 뜻이다.
      //
      // 즉: Scan → Filter 가 아니라 Filtered Scan 이 된다.
      notFromLGA.explain

      def main(args: Array[String]): Unit = {
        // 실행 로직 없음
      }
    }

8. Tungsten

  1. Tungsten

    : Catalyst 가 계획을 최적화하는 컴파일러라면, Tungsten 은 실행을 빠르게 만드는 저수준 엔진 최적화 레이어

    SQL / DF 코드
        ↓
    Logical Plan
        ↓
    Optimized Plan
        ↓
    Physical Plan
        ↓
    WholeStageCodegen (Tungsten)
        ↓
    컴파일된 Java 코드 실행
    • 초기 Spark 는 JVM 객체 기반으로 Java GC 에 의존했고, 객체 생성이 많아서 메모리 낭비가 많았다.
    • 그래서 GC Pause 가 발생하고, 메모리 단편화가 발생하고, CPU 캐시가 비효율적으로 사용되었다.
    • 그래서 Spark 는 JVM 객체 중심 구조를 버리고, 메모리를 직접 다루는 구조로 변경했다.

  2. Tungsten 의 핵심 요소

    • Off-Heap Memory Management
      • 기존 : JVM Heap 에 객체 생성 → GC 가 관리
      • Tungsten : Unsafe API 사용 → 직접 메모리 관리, JVM GC 최소화
        • GC 감소, 메모리 효율 증가, 큰 데이터 처리 안정성 증가
    • Cache-aware Computation
      • Spark 는 데이터를 객체로 저장하지 않고, 바이너리 형식(UnsafeRow)으로 저장하는데, 이 구조는 CPU 캐시에 더 잘 맞고, 메모리 접근이 연속적이며, 포인터 추적이 없음 → 즉, 메모리 접근 패턴 최적화
    • Whole-Stage Code Generation*

      • Catalyst 가 Physical Plan 을 만들면, Tungsten 은 전체 연산 파이프라인을 하나의 Java 코드로 컴파일한다.

        // 기존
        Scan → Filter → Project → Aggregate

        // 텅스텐 이후
        for (row in input) {
          if (조건) {
            결과 계산
          }
        }

        하나의 for-loop로 컴파일된다. → 함수 호출 감소, 객체 생성 감소, CPU 효율 증가

  3. Tungsten 이 없다면..

    • 없다면..
      • RDD 방식 : map → filter → reduce
        • 각 단계마다 객체 생성, 각 단계마다 함수 호출 → GC 증가
      • DF/DS + Tungsten : 연산 통합
        • 객체 최소화, byte array 기반 처리, 컴파일된 코드 실행
    • 있기 때문에..

      • UnsafeRow, Off-heap memory, WholeStageCodegen, CPU cache 최적화

        → 이 모든 것이 Tungsten 덕분이다.

  4. UnsafeRow

    : Spark 내부 Row 표현 방식로 바이너리 포맷이다. 컬럼을 byte 배열처럼 저장한다.

    • 객체 없음, 포인터 없음, 연속 메모리, 빠른 접근

  5. 실습

    package part2foundations

    import org.apache.spark.sql.SparkSession

    object TungstenDemo {

      // SparkSession 생성
      // local 모드에서 실행
      val spark = SparkSession.builder()
        .appName("Tungsten Demo")
        .master("local")
        .getOrCreate()

      // SparkContext 참조
      val sc = spark.sparkContext

      // ------------------------------------------------------------
      // 1. RDD 캐싱
      // ------------------------------------------------------------

      // 1 ~ 10,000,000까지 숫자를 RDD로 생성
      // cache() → 메모리에 저장 (JVM 객체 형태)
      val numbersRDD = sc.parallelize(1 to 10000000).cache()

      // 첫 번째 count()
      // → 실제 연산 실행
      // → 메모리에 캐시 저장
      numbersRDD.count()

      // 두 번째 count()
      // → 캐시에서 읽기
      // → 훨씬 빠름
      numbersRDD.count()

      /*
        여기서 중요한 점:

        RDD 캐시는 JVM 객체 형태로 저장됨.
        즉:
          - 객체 기반
          - GC 영향 받음
          - 메모리 효율 상대적으로 낮음
      */

      // ------------------------------------------------------------
      // 2. DataFrame 캐싱 (Tungsten 적용)
      // ------------------------------------------------------------

      import spark.implicits._

      // RDD → DataFrame 변환
      // cache() → Tungsten 기반 캐시 (UnsafeRow 형태)
      val numbersDF = numbersRDD.toDF("value").cache()

      // 첫 실행 → 실제 연산 + 캐시 저장
      numbersDF.count()

      // 두 번째 실행 → 캐시 사용
      numbersDF.count()

      /*
        여기서 핵심:

        DataFrame 캐시는
        - UnsafeRow (바이너리 포맷)
        - 객체 최소화
        - 메모리 효율 ↑
        - GC 감소

        즉, RDD 캐시보다 효율적이다.
      */

      // ------------------------------------------------------------
      // 3. WholeStageCodegen OFF
      // ------------------------------------------------------------

      /*
        WholeStageCodegen은 Tungsten의 핵심 기능.
        여러 연산을 하나의 컴파일된 코드로 합쳐 실행한다.

        아래 설정은 Codegen을 끄는 설정.
      */
      spark.conf.set("spark.sql.codegen.wholeStage", "false")

      val noWholeStageSum = spark.range(1000000).selectExpr("sum(id)")

      // 실행 계획 출력
      noWholeStageSum.explain()

      /*
        Physical Plan 예시 (Codegen OFF):

        HashAggregate
          HashAggregate
            Range

        * 표시가 없음.
        → Tungsten WholeStageCodegen 비활성 상태
      */

      noWholeStageSum.show()

      // ------------------------------------------------------------
      // 4. WholeStageCodegen ON
      // ------------------------------------------------------------

      // Codegen 활성화
      spark.conf.set("spark.sql.codegen.wholeStage", "true")

      val wholeStageSum = spark.range(1000000).selectExpr("sum(id)")

      wholeStageSum.explain()

      /*
        Physical Plan 예시 (Codegen ON):

        *(1) HashAggregate
        +- *(1) HashAggregate
           +- *(1) Range

        여기서 중요한 점:

        앞에 붙은 "*"

        → WholeStageCodegen 적용됨
        → 연산들이 하나의 파이프라인으로 합쳐짐
        → 하나의 for-loop로 컴파일됨
      */

      wholeStageSum.show()

      // ------------------------------------------------------------
      // main
      // ------------------------------------------------------------

      def main(args: Array[String]): Unit = {
        // Spark UI를 보기 위해 프로그램을 잠시 유지
        Thread.sleep(1000000)
      }
    }

1주차: Spark Caching과 Checkpointing의 차이와 실전 활용법

February 22, 2026 · 4 min read
Kwon Younglin
Data Engineer

스파크(Spark) 애플리케이션을 개발하다 보면 Join이나 정렬(Sort) 같은 무거운(Expensive) 연산 결과를 여러 번 재사용해야 하는 상황이 반드시 발생한다. 이때 아무 처리도 하지 않으면 액션(Action)이 호출될 때마다 데이터를 처음부터 다시 읽고 계산하는 엄청난 비효율이 발생한다.

이 문제를 해결하는 두 가지 핵심 무기가 바로 캐싱(Caching) 과 체크포인팅(Checkpointing) 이다. 두 기법은 비슷해 보이지만 내부 동작 원리와 목적이 완전히 다르다. 코드를 통해 이 둘의 차이를 확인한다.

1. Caching: 빠른 재사용을 위한 메모리 최적화​

캐싱은 계산이 끝난 DataFrame이나 RDD를 메모리나 디스크에 임시로 저장해 두는 기법이다. 가장 큰 특징은 데이터의 계산 족보(Lineage)를 그대로 유지한다는 점이다. 즉, 캐시된 데이터가 날아가면 스파크는 언제든 처음부터 다시 계산해서 복구할 수 있다.

1-1. DataFrame 캐싱과 StorageLevel​

스파크는 상황에 맞게 데이터를 어디에, 어떻게 저장할지 결정하는 다양한 StorageLevel을 제공한다.

import org.apache.spark.sql.SparkSession
import org.apache.spark.storage.StorageLevel

val spark = SparkSession.builder()
  .appName("CachingDemo")
  .config("spark.memory.offHeap.enabled", "true")
  .config("spark.memory.offHeap.size", 10000000)
  .master("local")
  .getOrCreate()

val flightsDF = spark.read.option("inferSchema", "true").json("src/main/resources/data/flights")
val orderedFlightsDF = flightsDF.orderBy("dist") // 무거운 연산 가정

// 다양한 캐싱 옵션 적용
orderedFlightsDF.persist(
  // 1. 메모리에 원본 객체로 저장 (가장 빠름, OOM 위험)
  // StorageLevel.MEMORY_ONLY 
  
  // 2. 디스크에만 저장 (메모리 절약, I/O로 인해 느림)
  // StorageLevel.DISK_ONLY 
  
  // 3. 메모리에 먼저 채우고, 넘치면 디스크로 Spill (DF 기본값)
  // StorageLevel.MEMORY_AND_DISK 

  // 4. 가비지 컬렉터(GC)를 회피하는 Off-Heap 메모리 저장 (Tungsten 엔진 활용)
  StorageLevel.OFF_HEAP 
)

orderedFlightsDF.count() // 첫 번째 액션: 계산 후 캐시 메모리에 적재됨
orderedFlightsDF.count() // 두 번째 액션: 캐시에서 바로 읽어와서 훨씬 빠름!

// 캐시 메모리 해제
orderedFlightsDF.unpersist()

1-2. RDD 직렬화 캐싱 (메모리 다이어트)​

오래된 RDD API를 사용할 때는 _SER(Serialized) 옵션이 필수적이다.

val flightsRDD = orderedFlightsDF.rdd

// 자바 객체 대신 바이트 배열로 직렬화하여 메모리 용량 극적 절감
flightsRDD.persist(StorageLevel.MEMORY_ONLY_SER)
flightsRDD.count()

RDD는 무거운 JVM 객체를 생성하기 때문에 MEMORY_ONLY를 쓰면 OOM이 터지기 쉽다. 하지만 DataFrame을 쓰면 스파크 카탈리스트(Catalyst) 엔진이 알아서 효율적인 바이너리 포맷으로 저장해주기 때문에 굳이 RDD로 변환해서 캐싱할 필요는 없다.

2. Checkpointing: DAG를 끊어내는 단두대​

체크포인팅은 메모리 부족이나 노드 장애를 대비해, 데이터를 임시가 아닌 HDFS나 로컬 디스크 같은 영구적이고 안전한 저장소에 물리적으로 기록해 버리는 기법이다.

가장 중요한 차이는 계산 족보(Lineage, DAG)를 싹둑 끊어버린다는 것이다.

2-1. Checkpointing 코드 구현​

체크포인팅을 사용하려면 반드시 스파크 컨텍스트(sc)에 저장할 디렉토리를 먼저 지정해야 한다.

val sc = spark.sparkContext
// 체크포인트 저장 경로 설정 (실무에서는 HDFS나 S3 경로 사용)
sc.setCheckpointDir("spark-warehouse")

val orderedFlights = flightsDF.orderBy("dist")

// checkpoint() 자체가 액션(Action)을 트리거하며 디스크에 데이터를 쓴다.
val checkpointedFlights = orderedFlights.checkpoint() 

checkpointedFlights.count() // 디스크에서 바로 읽어옴

2-2. 실행 계획(Physical Plan) 비교 분석​

캐싱과 체크포인팅의 근본적인 차이는 explain()을 통해 실행 계획을 열어봤을 때 명확하게 드러난다.

[일반 DF 또는 캐싱된 DF의 실행 계획]

orderedFlights.explain()
/*
== Physical Plan ==
*(1) Sort [dist#16 ASC NULLS FIRST], true, 0
+- *(1) Project [_id#7, arrdelay#8, ... ]
   +- BatchScan[...] JsonScan Location: InMemoryFileIndex[...]
*/

위처럼 캐시를 쓰더라도 스파크는 "여차하면 내가 파일부터 다시 읽어서 Sort할게"라며 전체 계획(Lineage)을 길게 들고 있다.

[체크포인트된 DF의 실행 계획]

checkpointedFlights.explain()
/*
== Physical Plan ==
*(1) Scan ExistingRDD[_id#7,arrdelay#8, ... ]
*/

체크포인트 이후의 데이터프레임은 이전의 Sort, Project, JsonScan 같은 복잡한 과거를 완전히 잊어버린다. 오직 디스크에 저장된 결과물(ExistingRDD)을 새로운 시작점(Source)으로 삼는다.

3. 요약 및 실전 결론: 언제 무엇을 써야 할까?​

🚀 Caching을 써야 할 때 (Performance Focus)​

  • 동일한 잡(Job) 안에서 머신러닝 모델을 반복 학습시킬 때.
  • 룩업 테이블(Lookup Table)처럼 자주 조인되는 작은 DataFrame을 재사용할 때.
  • 연산 속도를 쥐어짜야 하는 상황일 때.

🛡️ Checkpointing을 써야 할 때 (Reliability Focus)​

  • 배치 작업 시간이 너무 길어서 중간에 에러가 났을 때 처음부터 다시 돌리는 것이 재앙인 경우.
  • DataFrame의 withColumn이나 join 변환이 수십, 수백 번 겹쳐서 실행 계획(DAG)이 너무 길어지고 메모리를 많이 먹어 드라이버에서 StackOverflowError가 터질 위험이 있을 때. (이때 체크포인트를 한 번씩 찍어주면 DAG가 초기화되어 에러를 막을 수 있다.)
  • Spark Structured Streaming 운영 시 장애 복구를 위해.

💡 실무 운영 팁​

현업에서는 긴 데이터 파이프라인을 설계할 때, 중간중간 무거운 셔플(Shuffle)이 끝나는 지점에 캐싱과 체크포인팅을 함께 사용하기도 한다. 데이터를 메모리에 올려 빠르게 재사용하면서도, 디스크에 안전하게 백업해두어 DAG를 자르고 장애에 대비하는 훌륭한 전략이다.

1주차: Spark 내부 동작 원리 - 직렬화, Tungsten, Catalyst의 이해

스파크(Spark) 강의를 보다가 어떤 두가지 df를 repartition(7), repartition(9)으로 만든 후 join을 시도하는 예제를 봤다. 성능과 관련없이 repartition할 때와 join할 때에 어떤 explain()이 이루어지는지 확인하는 예제였지만 실제로 join전에 repartition을 한 번 해준다면 비효율적인지 궁금해졌다. 또한 최적화를 위해 텅스텐 엔진과 카탈리스트에 대한 이해가 필요해 이 글에서 설명한다.

1. 조인(Join) 전 Repartitioning, 과연 성능에 도움이 될까?​

1,000만 건의 데이터와 2,000만 건의 데이터를 조인(Join)해야 하는 상황을 가정해 보자. 이때 "조인 전에 미리 repartition(10) 정도로 파티션을 잘게 또는 균등하게 쪼개두면 병렬 처리가 잘 돼서 빠르지 않을까?"라고 생각할 수 도 있겠다.

결론부터 말하자면, 특별한 목적(Join Key 기반의 파티셔닝)이 없다면 조인 전의 단순한 repartition()은 성능을 높여주지 않는다.

조인과 셔플 파티션의 관계​

스파크에서 두 데이터를 조인할 때는 반드시 셔플(Shuffle) 이 발생한다. 스파크는 같은 조인 키(Key)를 가진 데이터를 같은 파티션(노드)으로 모으기 위해 데이터를 네트워크 너머로 재분배하는데, 이때 기본적으로 spark.sql.shuffle.partitions 설정값(기본값 200)에 따라 데이터를 200개의 파티션으로 쪼갠다.

만약 조인 전에 repartition(10)을 한다면 스파크는 다음과 같이 동작한다.

  1. 아무런 기준 없이 라운드 로빈(Round-Robin) 방식으로 데이터를 10개의 파티션으로 섞는다. (1차 셔플 발생 - 네트워크 I/O 낭비)
  2. 조인 연산을 만나면 조인 키를 기준으로 다시 데이터를 200개의 파티션으로 섞는다. (2차 셔플 발생)

즉, 의미 없는 셔플을 한 번 더 유발하여 성능만 떨어뜨리게 된다. 어차피 조인이 발생하면 스파크가 알아서 조인 키를 기준으로 200개로 리파티셔닝을 해주기 때문에, 스파크 엔진을 믿고 그대로 조인하는 것이 가장 빠르다.

2. API별 극단적 성능 차이: 직렬화(Serialization)​

같은 10억 건의 데이터를 세는(count()) 작업인데, 왜 API에 따라 16초가 걸리기도 하고 0.1초 만에 끝나기도 할까? 아래 코드를 통해 원인을 분석해 본다.

val sc = spark.sparkContext
// 1. RDD를 생성하고 DataFrame으로 변환 후 Count
val rdd = sc.parallelize(1 to 1000000000)
val df = rdd.toDF("id")
df.count() // 약 16초 소요 (매우 느림)

// 2. 처음부터 Dataset/DataFrame으로 생성 후 Count
val ds = spark.range(1, 1000000000)
ds.count() // 0.1초 소요 (매우 빠름)

16초 vs 0.1초 차이 나는 이유​

  • sc.parallelize를 사용하면 10억 개의 무거운 java.lang.Integer JVM 객체 가 메모리(Heap)에 생성된다. 이를 DataFrame으로 변환(toDF)하려면, 스파크는 10억 개의 객체를 일일이 꺼내서 스파크의 내부 바이너리 포맷으로 직렬화(SerializeFromObject) 하는 엄청난 노가다를 해야 한다. 이 변환 비용이 16초나 걸린 것이다.
  • 반면 spark.range는 애초에 JVM 객체를 만들지 않는다. 처음부터 텅스텐(Tungsten) 엔진이 좋아하는 바이너리 포맷으로 메모리에 데이터를 밀어 넣기 때문에 0.1초 만에 연산이 끝난다.

람다(Lambda) 함수​

Dataset은 빠르지만, 스칼라의 익명 함수(람다)를 쓰게되면 성능 저하가 일어난다.

// DS의 각 요소에 5를 곱하는 연산
val dsTimes5 = ds.map(_ * 5)
dsTimes5.selectExpr("count(*)").show() // 갑자기 7초로 느려짐 (70배 증가)

실행 계획(explain())을 보면 이유를 명확히 알 수 있다. 스파크 엔진 입장에서 _ * 5 라는 스칼라 코드는 속을 알 수 없는 블랙박스다. 최적화된 바이너리 데이터를 연산하기 위해 어쩔 수 없이 다시 JVM 객체로 역직렬화(DeserializeToObject)하고, 연산이 끝나면 다시 직렬화(SerializeFromObject)해야 한다.

💡 해결방법: 가능하면 ds.map(_ * 5) 대신, 스파크가 이해할 수 있는 컬럼 표현식인 df.select($"id" * 5)를 사용하면 직렬화 비용 없이 텅스텐 엔진으로 처리되기 때문에 결과는 같지만 직렬화 과정 없이 즉시 연산된다. 한 번 DataFrame을 선택했다면 끝까지 DataFrame의 DSL(내장 함수)을 유지하는 것이 핵심이다.

3. 텅스텐(Tungsten) 엔진과 Whole-Stage CodeGen​

텅스텐(Tungsten)은 JVM의 한계를 극복하고 하드웨어(CPU, 메모리)의 성능을 극한까지 끌어올리기 위한 스파크의 핵심 엔진이다. 텅스텐 엔진이 발동되는 가장 대표적인 최적화 기술이 바로 전체 스테이지 코드 생성(Whole-Stage Code Generation)이다.

아래 코드를 통해 텅스텐 엔진을 껐다 켰을 때의 차이를 살펴보자.

// 1. 텅스텐 엔진 강제 끄기
spark.conf.set("spark.sql.codegen.wholeStage", "false")
val noWholeStageSum = spark.range(1000000).selectExpr("sum(id)")
noWholeStageSum.explain()
/*
== Physical Plan ==
HashAggregate(keys=[], functions=[sum(id#54L)])
+- HashAggregate(keys=[], functions=[partial_sum(id#54L)])
   +- Range (0, 1000000, step=1, splits=1)
*/

// 2. 텅스텐 엔진 켜기 (기본값)
spark.conf.set("spark.sql.codegen.wholeStage", "true")
val wholeStageSum = spark.range(1000000).selectExpr("sum(id)")
wholeStageSum.explain()
/*
== Physical Plan ==
*(1) HashAggregate(keys=[], functions=[sum(id#67L)])
+- *(1) HashAggregate(keys=[], functions=[partial_sum(id#67L)])
   +- *(1) Range (0, 1000000, step=1, splits=1)
*/

실행 계획의 별표(*)가 의미하는 것​

텅스텐 엔진을 켜고 explain()을 호출하면 노드 앞에 *(1) 이라는 별표가 붙는다. 이것이 바로 Tungsten이 개입하여 Whole-Stage CodeGen이 적용되었다는 표시이다.

4. 카탈리스트(Catalyst): 쿼리 플랜 최적화​

텅스텐이 물리적인 연산 속도를 높인다면, 카탈리스트(Catalyst) 옵티마이저는 쿼리의 논리적인 구조 자체를 가장 효율적인 경로를 찾아준다.

val flights = spark.read.option("inferSchema", "true").json("src/main/resources/data/flights")

// 의사소통 부재로 인해 똑같은 데이터를 필터링하는 중복 코드가 발생한 상황
def filterTeam1(flights: DataFrame) = flights.where($"origin" =!= "LGA").where($"dest" === "DEN")
def filterTeam2(flights: DataFrame) = flights.where($"origin" =!= "EWR").where($"dest" === "DEN")

val filterBoth = filterTeam1(filterTeam2(flights))
filterBoth.explain(true)

여러 개발자가 협업하다 보면 위처럼 비효율적인 필터링(where)이 중첩될 수 있다. 하지만 카탈리스트는 물리적 실행 계획을 짜기 전에 이 논리들을 분석하여 중복을 제거하고 조건들을 하나의 깔끔한 식으로 합쳐준다.

파티션/필터 푸시다운 (Pushdown)​

카탈리스트의 가장 무서운 점은 소스(Parquet 등)에서 데이터를 읽을 때 발휘된다.

val notFromLGA = spark.read.load("src/main/resources/data/flights_parquet")
  .where($"origin" =!= "LGA")

이 코드를 실행하면, 스파크는 메모리에 데이터를 다 올려놓고 필터링을 하는 것이 아니라, 디스크(Parquet)에서 데이터를 읽어 들일 때부터 이미 "LGA"가 아닌 데이터만 골라서 읽어온다. 이를 필터 푸시다운(Filter Pushdown)이라고 하며, 불필요한 I/O를 원천 차단하여 성능을 기하급수적으로 높여준다.

📝 최종 결론 (Best Practices)​

  1. 조인 연산 전 의미 없는 repartition()은 성능저하를 불러올 수 있다.
  2. 스파크 코드를 짤 때는 무거운 객체를 만드는 RDD나 ds.map() 같은 람다 함수 사용을 피하고, 가급적 DataFrame의 내장 함수(DSL)를 사용해 직렬화를 피하는 것이 최적화 팁이다.
  3. 스파크는 텅스텐(Whole-Stage CodeGen)과 카탈리스트(Pushdown)라는 빠른 자체 엔진을 가지고 있다. 불필요햐게 직렬화를 하지 않고 텅스텐 엔진을 최대한 활용하는 코드 작성하는 것이 필요하다.

  **// 함수형 프로그램의 목표 = 불변성 + 순수 함수**

  // =========================

  // =========================
  val anIfExpression = if(2 > 3) "bigger" else "smaller"
  // Scala에서 if는 표현식(expression)
  // 즉, 값을 반환함
  // 2 > 3 은 false이므로 결과는 "smaller"
  // 자바와 달리 if가 값이 있다는 점이 중요

  // =========================
  // 3. 명령(instructions) vs 표현(expressions)
  // =========================
  val theUnit = println("Hello, Scala")

  // 스칼라에서는 뭐든 반환을 할 수 있도록 해서 unit 을 반환 (void 반환 가능)
  // Unit은 "의미 있는 값이 없음"을 의미 (자바의 void와 유사)
  // 하지만 Scala에서는 Unit도 하나의 값 -> val x: Unit = ()

  // =========================
  // 4. 함수(functions) 기본
  // =========================
  def myFunction(x: Int) = 42
  // def는 함수 정의
  // x는 Int 타입
  // 반환 타입은 생략했지만 Int로 추론됨
  // 입력과 무관하게 항상 42 반환

  // =========================

  // =========================
  class Animal
  // 기본 클래스 정의

  class Cat extends Animal
  // Animal을 상속

  trait Carnivore {
    def eat(animal: Animal): Unit
  }
  // trait는 자바의 interface와 유사 / eat 메서드는 구현 없이 선언만 함

  class Crocodile extends Animal with Carnivore {
    override def eat(animal: Animal): Unit = println("Crunch!")
  }
  // extends는 클래스 상속
  // with는 trait를 섞는(mixin) 것 / override 키워드로 trait의 메서드 구현

  // =========================
  // 13. try-catch (표현식)
  // =========================
  try {
    throw new NullPointerException
  } catch {
    case _: NullPointerException => "some returned value"
    case _: Throwable => "something else"
  }
  // try-catch도 표현식
  // 예외 타입에 따라 값 반환 가능
  // 여기서는 NPE 발생 → "some returned value"
  // 그리고 해당하는 조건 하나만 무조건 탐 <- break 필요 없음

  // =========================
  // 14. Future (비동기)
  // =========================
  import scala.concurrent.ExecutionContext.Implicits.global
  // Future 실행에 필요한 스레드 풀 제공 -> **global은 기본 스레드 풀 제공**

  val aFuture = Future {
    // 다른 스레드에서 실행되는 블록
    42
  }
  // aFuture는 Future[Int]

  aFuture.onComplete {
    case Success(meaningOfLife) =>
      println(s"I've found $meaningOfLife")
    case Failure(ex) =>
      println(s"I have failed: $ex")
  }
  // onComplete 은 Future 완료 시의 콜백
  // 성공이면 Success / 실패하면 Failure

  // =========================
  // 15. 부분 함수(Partial functions)
  // =========================
  val aPartialFunction: PartialFunction[Int, Int] = {
    case 1 => 43
    case 8 => 56
    case _ => 999
  }
  // PartialFunction은 일부 입력에만 정의될 수 있는 함수
  // 여기서는 _ => 999가 있어서 사실상 모든 입력 처리
  // 만약 1/8 조건만 있었을 경우, 1/8만 처리하는 거라 부분 함수겠지만, _가 전체 함수 역할

  // =========================
  // 16. 암시적 기능(Implicits)
  // =========================
  // 16-1. 암시적 인자(implicit parameter)
  def methodWithImplicitArgument(implicit x: Int) = x + 43
  // implicit 키워드가 붙은 파라미터는
  // 호출 시 자동으로 주입될 수 있음

  implicit val implicitInt = 67
  // 스코프 내에 implicit Int 정의

  val implicitCall = methodWithImplicitArgument
  // 명시적으로 인자를 넘기지 않았지만
  // implicitInt가 자동으로 전달됨
  // 결과: 67 + 43 = 110

  // 16-2. implicit conversion (implicit def)
  case class Person(name: String) {
    def greet = println(s"Hi, my name is $name")
  }
  // case class는 자동으로 equals, toString, apply 등 생성

  implicit def fromStringToPerson(name: String) = Person(name)
  // String을 Person으로 암시적 변환

  "Bob".greet
  // 원래 String에는 greet가 없음
  // 컴파일러가 fromStringToPerson("Bob")로 변환 후 greet 호출

  // 16-3. 확장 메서드(implicit class)
  implicit class Dog(name: String) {
    def bark = println("Bark!")
  }
  // String에 bark 메서드를 추가한 것처럼 동작

  "Lassie".bark

  /*
    implicit 탐색 범위
    - local scope (현재 스코프)
    - imported scope (import된 영역)
    - companion objects of the types involved
      (관련 타입의 companion object 내부)

    컴파일러는 이 순서로 implicit을 탐색함
   */

}

package part1recap
// 패키지 선언

import org.apache.spark.rdd.RDD
// Spark의 저수준 API인 RDD 타입을 쓰기 위한 import

import org.apache.spark.sql.SparkSession
// Spark SQL/Structured API의 진입점인 SparkSession

import org.apache.spark.sql.functions._
// col, expr, avg 같은 컬럼/함수 유틸을 쓰기 위한 import
// (언더스코어 * 처럼 모든 함수를 가져오는 방식)

object SparkRecap {
  // object: 싱글톤. 이 파일에서 Spark 관련 코드를 담는 실행 단위
  // (여기서는 main이 아래에 있으므로, main 실행 시점에 프로그램이 시작됨)

  // =========================
  // 1) SparkSession 생성
  // =========================
  // Spark structured API 의 entry point
  val spark = SparkSession.builder()
    // SparkSession을 설정하는 빌더(builder) 패턴 시작
    .appName("Spark Recap")
    // 애플리케이션 이름(로그/웹 UI 등에서 보임)
    .master("local[2]")
    // 실행 모드 지정
    // -> local[2] = 로컬 머신에서 쓰레드 2개로 실행(로컬 테스트용)
    //    실제 클러스터면 여기가 달라짐
    .getOrCreate()
    // 기존에 같은 설정으로 떠 있는 SparkSession이 있으면 재사용, 없으면 새로 생성

  // =========================
  // 2) DataFrame 읽기 (cars)
  // =========================
  // read a DF
  val cars = spark.read
    // DataFrameReader: 다양한 포맷(JSON/CSV/Parquet 등) 읽는 진입점
    .format("json")
    // 읽을 파일 포맷 지정
    .option("inferSchema", "true")
    // 스키마를 자동 추론하도록 설정(느릴 수 있음, 운영에서는 스키마 명시 권장)
    .load("src/main/resources/data/cars")
    // 파일/폴더 경로에서 로드
    // JSON은 보통 "폴더"를 주면 그 안의 파일들을 읽음

  // =========================
  // 3) spark.implicits import
  // =========================
  import spark.implicits._
  // Spark의 암시적 기능(implicits)을 가져옴 <- 컴파일러가 자동으로 해주는 변환
  // 대표적으로
  // - $"colName" 문법(컬럼을 Column 객체로)
  // - toDF, toDS, as[T] 같은 변환
  // - Dataset Encoder (JVM 객체 ↔ Spark 내부 Row 변환기) 자동 생성
  // 이런 기능들이 활성화됨

  // =========================
  // 4) select (컬럼 선택/변환)
  // =========================
  // select
  val usefulCarsData = cars.select(
    col("Name"), // column object
    $"Year", // another column object (needs spark implicits)
    (col("Weight_in_lbs") / 2.2).as("Weight_in_kg"),
    expr("Weight_in_lbs / 2.2").as("Weight_in_kg_2")
  )
  
  // col("Name")은 Column 객체를 만들기 위한 함수 <- functions._ import 덕분에 사용 가능
  // $"Year"는 spark.implicits._ 가 있어야 가능한 문법 <- 문자열을 Column으로 바꿔주는 확장 문법
  // Weight_in_lbs 컬럼을 2.2로 나눠서 kg로 바꾸고 <- 결과 컬럼 이름을 Weight_in_kg로 지정
  // expr는 SQL 문자열 표현식을 Column으로 해석해줌 <- 같은 계산이지만 "문자열 SQL"로 적는 방식

  val carsWeights = cars.selectExpr("Weight_in_lbs / 2.2")
  // selectExpr는 여러 개의 SQL 표현식을 문자열로 넘길 수 있는 select의 변형
  // 결과는 DataFrame(스키마 기반)이며 컬럼 이름은 표현식 기반으로 자동 생성될 수 있음

  // =========================
  // 5) filter / where
  // =========================
  // filter
  val europeanCars = cars.where(col("Origin") =!= "USA")
  // where는 filter와 거의 동일
  // =!= 는 "같지 않다" 연산(컬럼 비교용)
  // 즉 Origin이 USA가 아닌 행만 남김

  // =========================
  // 6) aggregations (집계)
  // =========================
  // aggregations
  val averageHP = cars.select(avg(col("Horsepower")).as("average_hp"))
  // avg로 Horsepower 평균을 구하고 컬럼명을 average_hp로
  // (주석의 sum, mean, stddev, min, max 등도 같은 계열 집계 함수)

  // =========================
  // 7) grouping (groupBy)
  // =========================
  // grouping
  val countByOrigin = cars
    .groupBy(col("Origin")) // a RelationalGroupedDataset
    // groupBy 결과는 "그룹 연산을 할 수 있는 중간 객체" (RelationalGroupedDataset)
    .count()
    // 그룹별 레코드 개수 집계
    // 결과는 Origin별 count 컬럼을 가진 DataFrame

  // =========================
  // 8) join (조인)
  // =========================
  // joining
  val guitarPlayers = spark.read
    .option("inferSchema", "true")
    .json("src/main/resources/data/guitarPlayers")
    // format("json") 대신 .json(path) 단축 사용 가능

  val bands = spark.read
    .option("inferSchema", "true")
    .json("src/main/resources/data/bands")

  val guitaristsBands = guitarPlayers.join(
    bands,
    guitarPlayers.col("band") === bands.col("id")
  )
  // join 조건
  // guitarPlayers의 band 컬럼 == bands의 id 컬럼
  // === 는 Column 간 "같다" 비교(동등 비교)
  // 기본 join type은 inner join

  /*
    join types
    - inner: only the matching rows are kept
      (매칭되는 행만 유지)
    - left/right/full outer join
      (한쪽/양쪽 모두의 미매칭도 포함)
    - semi/anti
      (존재 여부만 필터링: semi는 매칭 있는 쪽만, anti는 매칭 없는 쪽만)
   */

  // =========================
  // 9) Dataset (타입 있는 분산 컬렉션)
  // =========================
  // datasets = typed distributed collection of objects
  case class GuitarPlayer(id: Long, name: String, guitars: Seq[Long], band: Long)
  // Dataset에서 사용될 타입(스키마)을 case class로 정의
  // case class는 자동으로 생성자/equals/toString 등 제공

  val guitarPlayersDS = guitarPlayers.as[GuitarPlayer] // needs spark.implicits
  // DataFrame(스키마 기반 Row 컬렉션) -> Dataset[GuitarPlayer] (타입 기반)
  // 여기서 Encoder가 필요하며 spark.implicits._ 가 있어야 자동 제공되는 경우가 많음

  guitarPlayersDS.map(_.name)
  // Dataset은 타입이 있으니 map에서 Row가 아니라 GuitarPlayer 객체로 다룸
  // _.name은 GuitarPlayer의 name 필드를 뽑음
  // 결과는 Dataset[String]

  // =========================
  // 10) Spark SQL (SQL로 조회)
  // =========================
  // Spark SQL
  cars.createOrReplaceTempView("cars")
  // DataFrame을 임시 뷰로 등록
  // 세션(spark) 범위에서만 유효한 "임시 테이블" 같은 개념

  val americanCars = spark.sql(
    """
      |select Name from cars where Origin = 'USA'
    """.stripMargin
  )
  // spark.sql로 SQL 문자열 실행
  // stripMargin은 멀티라인 문자열에서 | 기준으로 들여쓰기 제거
  // 결과는 DataFrame

  // =========================
  // 11) 저수준 API: RDD
  // =========================
  // low-level API: RDDs
  val sc = spark.sparkContext
  // SparkContext는 Spark의 기본 컨텍스트(클러스터 연결, RDD 생성 등)
  // SparkSession 안에 포함되어 있음

  val numbersRDD: RDD[Int] = sc.parallelize(1 to 1000000)
  // 1~1,000,000 범위를 RDD[Int]로 병렬화
  // parallelize는 드라이버에 있는 컬렉션을 분산 데이터로 만드는 방식(테스트용으로 좋음)
  // 대규모 운영 데이터는 보통 파일/테이블에서 읽어 생성

  // functional operators
  val doubles = numbersRDD.map(_ * 2)
  // RDD의 map 변환: 각 요소에 *2
  // 결과는 새로운 RDD[Int]
  // (RDD 변환도 lazy: action이 실행되기 전까지 실제 계산 안 됨)

  // =========================
  // 12) RDD <-> DF/DS 변환
  // =========================
  // RDD -> DF
  val numbersDF = numbersRDD.toDF("number") // you lose type info, you get SQL capability
  // RDD[Int]를 DataFrame으로 변환
  // DataFrame은 내부적으로 Dataset[Row]라서 "타입 정보"는 Row로 일반화됨(타입 안전성 감소)
  // 대신 Spark SQL 기능(컬럼 연산, 최적화, SQL)이 강력해짐
  // toDF는 spark.implicits._ 필요

  // RDD -> DS
  val numbersDS = spark.createDataset(numbersRDD)
  // RDD[Int] -> Dataset[Int]
  // Dataset은 타입을 유지하면서도 최적화(SQL 플래너) 일부 혜택을 받을 수 있음
  // (단, Dataset은 Scala/Java 쪽에서 주로 쓰고 PySpark는 보통 DataFrame 중심)

  // DS -> RDD
  val guitarPlayersRDD = guitarPlayersDS.rdd
  // Dataset[GuitarPlayer] -> RDD[GuitarPlayer]
  // 저수준 API로 내려가고 싶을 때 사용

  // DF -> RDD
  val carsRDD = cars.rdd // RDD[Row]
  // DataFrame -> RDD[Row]
  // Row는 스키마 기반의 일반 레코드(컬럼을 인덱스/이름으로 접근)

  // =========================
  // 13) main: 실행 진입점
  // =========================
  def main(args: Array[String]): Unit = {
    // showing a DF to the console
    cars.show()
    // DataFrame을 콘솔에 출력 (기본 20행, truncate 옵션 등 있음)

    cars.printSchema()
    // DataFrame 스키마 출력 (컬럼명/타입/nullable 여부)
  }
