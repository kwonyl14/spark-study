1주차: Memory, Caching and Checkpointing

February 22, 2026 · 9 min read
Hwang Haejeong
Data Engineer

1. Executor Memory Architecture

  1. Spark Executor 메모리

    • Execution Memory : joins, sorts, groups 등 보통 shuffle 연산에서 사용
    • Storage Memory : RDD 를 메모리에 캐싱하여 재사용 및 빠른 조회 목적으로 사용

  2. Spark 1.x vs Spark 1.6+ 메모리 모델

    • 과거 (Static Memory Management)
      • 메모리를 미리 나눠놓았다. (Execution Memory / Storage Memory) → 고정 비율
      • 문제점 : Execution 이 부족해도 Storage 를 못 가져오고, Storage 가 부족해도 Execution 못 써서 비효율적
    • 현재 (Unified Memory Management)
      • Spark 1.6 이후부터는 Execution과 Storage가 서로 빌려 쓸 수 있게 되어 훨씬 유연해졌다.
        • Execution 이 메모리 부족하면 → Storage 영역 일부 사용 가능
        • Storage 가 꽉 차면 → LRU 알고리즘으로 eviction

  3. 메모리 구조

    설정의미
    spark.memory.fractionexecutor 메모리 중 unified 영역 비율
    spark.memory.storageFractionunified 영역 중 storage 비율

    RDD 2개가 캐시되어 있고, Storage 영역 일부 남아 있을 때

    큰 shuffle 연산이 실행되면 Storage 의 남는 부분을 Execution 이 빌려서 사용

    Storage 에 RDD가 너무 많고, Execution 영역 확보 필요할 때

    LRU 정책으로 오래된 RDD 제거하여 Execution 영역을 확보 ← 반드시 Execution 영역은 항상 확보가 되어야 한다.

  4. 설정 치트시트

    설정설명
    spark.executor.memoryexecutor 전체 메모리
    spark.memory.fractionunified 영역 비율
    spark.memory.storageFractionstorage 비율
    spark.memory.offHeap.sizeoff-heap 메모리 (Tungsten)
    spark.yarn.executor.memoryOverhead컨테이너 메모리 오버헤드
    spark.driver.memory드라이버 메모리

2. Caching

  1. Caching 이 필요한 이유

    • Spark 에서는 Action 이 실행될 때마다 Job 이 시작되는데,

      df.count()
      df.collect()

      → action 이 두 번 실행하면 lineage 전체를 두 번 계산하게 된다.

      → Cache expensive RDDs : lineage 는 유지하고, 계산된 데이터를 executor 메모리에 저장하고 이후 Job 에서 재사용할 수 있도록 해준다.

      Action 실행
          ↓
      Lineage 따라 계산
          ↓
      cache() 없으면 매번 재계산

      cache() 있으면
          ↓
      Executor 메모리에 저장
          ↓
      다음 Job에서 재사용

  2. 동작 원리

    Cache 는 Driver 에 저장되는 게 아니고, Worker 의 executor 메모리에 저장된다.

    • 캐시의 저장 위치
      • Heap memory, Off-heap memory(tungsten), Disk, Memory+Disk
    • JVM 주의사항

      • JVM heap 이 너무 크면 GC 시간이 증가하고, Full GC 멈춤 현상이 발생할 가능성이 높아지고 GC 가 길어지면서 성능이 저하된다. JVM heap 이 너무 작으면 OOM 이 발생하고, 캐시 저장이 불가능하게 된다.

        메모리를 크게 잡는 게 항상 좋은 건 아니다.

        • Executor 당 힙은 대개 816GB(많아도 2430GB 근처) 에서 시작
        • 정말 필요할 때만 키우고, 가능하면 32GB 넘기지 않게 운영 설계

  3. Persistence Level 종류

    옵션설명
    MEMORY_ONLY메모리에 비직렬화 저장
    MEMORY_AND_DISK메모리 부족 시 디스크 저장
    MEMORY_ONLY_SER직렬화 후 메모리 저장
    DISK_ONLY디스크에만 저장
    MEMORY_ONLY_22개 복제
    OFF_HEAPoff-heap 저장
    myRDD.cache()
    myRDD.persist(StorageLevel.MEMORY_ONLY) // -> 객체가 많고 GC 부담이 크다.
    myRDD.unpersist()

  4. Raw 와 Serialized 객체

    • Raw (기본 MEMORY_ONLY) : 메모리 사용량 큼, RAM 처리 매우 빠름, CPU 효율적
    • Serialized (MEMORY_ONLY_SER) : 메모리 효율 좋음, CPU 사용 증가, 디스크 읽기 빠름
    구분RawSerialized
    메모리많이 사용적게 사용
    CPU적게 사용많이 사용
    RAM 속도빠름느림
    Disk 속도느림빠름

  5. Fault Tolerance 노드 장애 발생 시 캐시 파티션이 손실되고, lineage 기반으로 재계산이 이뤄지는데.. 복제 옵션(MEMORY_ONLY_2 등)을 쓰면 일부 보호가 가능하다.

  6. Caching Recommendations

    • 많이 재사용되는 것만 캐시 ← 남발하면 OOM 위험
    • 데이터가 메모리에 충분히 들어가면 → MEMORY_ONLY
    • 메모리 부족하면 → MEMORY_ONLY_SER
    • 디스크 캐시는? → 간단한 연산은 디스크에서 읽는 것보다 재계산이 더 빠를 수 있다.

  7. 실습

    package part3caching

    import org.apache.spark.sql.SparkSession
    import org.apache.spark.storage.StorageLevel

    object Caching {

      // SparkSession 생성
      // off-heap 메모리 사용 설정 (Tungsten 기반)
      val spark = SparkSession.builder()
        .appName("Caching")

        // off-heap 메모리 사용 활성화
        // JVM heap 밖의 메모리를 사용 → GC 부담 감소
        .config("spark.memory.offHeap.enabled", "true")

        // off-heap 메모리 크기 설정 (bytes 단위)
        // 여기서는 약 10MB
        .config("spark.memory.offHeap.size", 10000000)

        // 로컬 모드 실행
        .master("local")
        .getOrCreate()

      // JSON 파일 읽기
      // inferSchema = true → 자동으로 타입 추론
      val flightsDF = spark.read
        .option("inferSchema", "true")
        .json("src/main/resources/data/flights")

      // Action 1
      // 캐시 없이 전체 데이터 계산
      flightsDF.count()

      // "비싼 연산" 시뮬레이션
      // orderBy는 shuffle 발생 → expensive operation
      val orderedFlightsDF = flightsDF.orderBy("dist")

      // 시나리오:
      // orderedFlightsDF를 여러 번 사용한다고 가정

      orderedFlightsDF.persist(

        // 기본값: MEMORY_AND_DISK
        // persist()에 인자를 안 주면 MEMORY_AND_DISK

        // StorageLevel.MEMORY_ONLY
        // → JVM heap 메모리에 비직렬화 상태로 저장
        // → CPU 효율적 (빠름)
        // → 메모리 많이 사용

        // StorageLevel.DISK_ONLY
        // → 디스크에만 저장
        // → 메모리 절약
        // → 읽을 때 느림

        // StorageLevel.MEMORY_AND_DISK
        // → 먼저 메모리에 저장
        // → eviction되면 디스크로 fallback

        /* 직렬화 옵션 */

        // StorageLevel.MEMORY_ONLY_SER
        // → 메모리에 직렬화 형태로 저장
        // → 메모리 절약
        // → CPU 사용 증가

        // StorageLevel.MEMORY_ONLY_2
        // → 메모리에 2개 복제
        // → 장애 대비
        // → 메모리 2배 사용

        // StorageLevel.MEMORY_ONLY_SER_2
        // → 직렬화 + 2배 복제

        /* off-heap 캐시 */

        StorageLevel.OFF_HEAP
        // → JVM heap 밖에 저장
        // → Tungsten 기반
        // → GC 영향 거의 없음
        // → 반드시 offHeap 설정 필요
      )

      // Action 2
      // 첫 번째 실행 → 실제 계산 발생 + 캐시 저장
      orderedFlightsDF.count()

      // Action 3
      // 두 번째 실행 → 캐시에서 읽음 → 매우 빠름
      orderedFlightsDF.count()

      // 캐시 제거
      // Executor 메모리에서 제거
      orderedFlightsDF.unpersist()

      // Temp View 생성
      orderedFlightsDF.createOrReplaceTempView("orderedFlights")

      // 테이블 단위 캐시
      // Spark SQL 레벨에서 캐시
      spark.catalog.cacheTable("orderedFlights")

      // SQL 캐시 사용
      orderedFlightsDF.count()

      // --------------------------
      // RDD 캐시 예제
      // --------------------------

      // DataFrame → RDD 변환
      val flightsRDD = orderedFlightsDF.rdd

      // RDD는 보통 MEMORY_ONLY_SER 많이 사용
      // RDD는 객체 기반이므로 직렬화가 메모리 절약에 효과적
      flightsRDD.persist(StorageLevel.MEMORY_ONLY_SER)

      // 첫 실행 → 계산 + 캐시 저장
      flightsRDD.count()

      def main(args: Array[String]): Unit = {
        // UI 관찰용 대기
        Thread.sleep(1000000)
      }
    }

3. Checkpointing

  1. Checkpointing vs Caching 

    • Caching : lineage(의존성 그래프) 유지, Executor의 Storage Memory 사용, 노드 장애 시 lineage 기반 → 재계산

      ⇒ 목적 : 성능 향상

    • Checkpointing : lineage 삭제, 외부 저장소(HDFS 등)에 저장, 노드 장애 시 → 디스크에서 로드

      ⇒ 목적 : 안정성 확보

    구분CachingCheckpoint
    저장 위치Executor 메모리외부 파일 시스템
    lineage유지삭제
    속도빠름느림
    장애 시재계산디스크에서 로드
    목적성능안정성
    • 장애 발생 시
      • Caching : Executor 장애 발생 → 캐시된 파티션 손실 → lineage 따라 해당 파티션만 재계산
      • Checkpointing : Executor 장애 발생 → checkpoint 파일에서 복구 → lineage 재계산 불필요

  2. Checkpointing

    체크 포인트는 외부 저장소에 저장하고, 리니지를 삭제한다. (DAG 단절)

    메모리를 사용하지 않기 때문에, 디스크 I/O 가 발생하여 캐시보다 느리고 더 많은 디스크 공간을 사용한다.

    • 언제 사용할지

      재계산 비용이 매우 크거나 / DAG 이 너무 길 때 / 점진적 누적 계산일 때 / Job 이 자주 실패할 때 / 실시간 데이터의 경우 사용하면 좋다.

      → 체크포인트를 통해 lineage가 끊기면서 이전 연산 결과가 메모리에 유지되지 않아도 되기 때문에, 긴 DAG 으로 인한 메모리 누적 부담이 감소되어 OOM 발생 가능성이 줄어든다.

      → 작업이 너무 느릴 경우(반복 계산)에 캐시를 통해 성능 개선을 하고..

  3. 실습

    package part3caching

    import org.apache.spark.sql.SparkSession
    import org.apache.spark.storage.StorageLevel

    object Checkpointing {

      // SparkSession 생성
      val spark = SparkSession.builder()
        .appName("Checkpointing")
        .master("local")
        .getOrCreate()

      // SparkContext 참조 (RDD용 + checkpoint 설정용)
      val sc = spark.sparkContext

      // ---------------------------------------------------------
      // 1. DataFrame Checkpoint 데모
      // ---------------------------------------------------------
      def demoCheckpoint() = {

        // JSON 파일 로드
        val flightsDF = spark.read
          .option("inferSchema", "true")
          .json("src/main/resources/data/flights")

        // 비용 큰 연산 (shuffle 발생)
        // orderBy는 Sort + Exchange → expensive
        val orderedFlights = flightsDF.orderBy("dist")

        // 체크포인트는 반드시 디렉토리 설정 필요
        // 외부 저장소 (보통 HDFS)
        sc.setCheckpointDir("spark-warehouse")

        // 체크포인트 실행
        // DataFrame을 디스크에 저장
        // lineage를 끊는다
        val checkpointedFlights = orderedFlights.checkpoint()

        // ------------------------------
        // 체크포인트 전 explain()
        // → 원래 JSON Scan + Sort 등 모든 lineage 보임
        // ------------------------------
        orderedFlights.explain()

        /*
          == Physical Plan ==
          Sort
            Project
              JsonScan
          → 원본 데이터에서 계산 시작
        */

        // ------------------------------
        // 체크포인트 후 explain()
        // → ExistingRDD로 바뀜
        // → 이제 디스크에서 읽음
        // ------------------------------
        checkpointedFlights.explain()

        /*
          == Physical Plan ==
          Scan ExistingRDD
          → lineage 삭제됨
          → 외부 저장된 데이터로부터 시작
        */

        checkpointedFlights.show()
      }

      // ---------------------------------------------------------
      // 2. RDD 캐싱 예제
      // ---------------------------------------------------------
      def cachingJobRDD() = {

        // 1~1천만 숫자 RDD 생성
        val numbers = sc.parallelize(1 to 10000000)

        // 내림차순 정렬 (shuffle 발생)
        val descNumbers = numbers
          .sortBy(-_)
          .persist(StorageLevel.DISK_ONLY)

        // 첫 실행 → 실제 계산 + 디스크 저장
        descNumbers.sum()

        // 두 번째 실행 → 디스크에서 읽음 → 더 빠름
        descNumbers.sum()
      }

      // ---------------------------------------------------------
      // 3. RDD 체크포인트 예제
      // ---------------------------------------------------------
      def checkpointingJobRDD() = {

        sc.setCheckpointDir("spark-warehouse")

        val numbers = sc.parallelize(1 to 10000000)

        val descNumbers = numbers.sortBy(-_)

        // RDD 체크포인트는 Unit 반환
        // 실제 저장은 Action 시점
        descNumbers.checkpoint()

        // 첫 실행 → 계산 + 디스크에 저장 + lineage 삭제
        descNumbers.sum()

        // 두 번째 실행 → checkpoint 파일에서 읽음
        descNumbers.sum()
      }

      // ---------------------------------------------------------
      // 4. DataFrame 캐싱 예제
      // ---------------------------------------------------------
      def cachingJobDF() = {

        val flightsDF = spark.read
          .option("inferSchema", "true")
          .json("src/main/resources/data/flights")

        val orderedFlights = flightsDF.orderBy("dist")

        // 디스크 캐시
        orderedFlights.persist(StorageLevel.DISK_ONLY)

        // 첫 실행 → 계산 + 디스크 저장
        orderedFlights.count()

        // 두 번째 실행 → 디스크에서 읽음 → 빠름
        orderedFlights.count()
      }

      // ---------------------------------------------------------
      // 5. DataFrame 체크포인트 예제
      // ---------------------------------------------------------
      def checkpointingJobDF() = {

        sc.setCheckpointDir("spark-warehouse")

        val flightsDF = spark.read
          .option("inferSchema", "true")
          .json("src/main/resources/data/flights")

        val orderedFlights = flightsDF.orderBy("dist")

        // DF 체크포인트
        val checkpointedFlights = orderedFlights.checkpoint()

        // 첫 실행 → 계산 + 저장 + lineage 삭제
        checkpointedFlights.count()

        // 두 번째 실행 → 디스크에서 읽음
        checkpointedFlights.count()
      }

      // ---------------------------------------------------------
      // 메인 실행
      // ---------------------------------------------------------
      def main(args: Array[String]): Unit = {

        // 캐시 실험
        cachingJobDF()

        // 체크포인트 실험
        checkpointingJobDF()

        // Spark UI 확인용 대기
        Thread.sleep(1000000)
      }
    }
