# Partitioning

**1. Repartition & Coalesce**

- **Repartition**
    - 데이터를 모든 파티션에 고르게 다시 분배한다.
    - 완전히 새로 섞어서 균등하게 만든다.
    - 원본 → repartition → 균등 분배된 상태

![스크린샷 2026-03-01 오후 2.27.17.png](Partitioning/%E1%84%89%E1%85%B3%E1%84%8F%E1%85%B3%E1%84%85%E1%85%B5%E1%86%AB%E1%84%89%E1%85%A3%E1%86%BA_2026-03-01_%E1%84%8B%E1%85%A9%E1%84%92%E1%85%AE_2.27.17.png)

- **Coalesce**
    - 기존 파티션을 붙여서 새로운 파티션을 만든다.
    - 전체를 다시 섞지 않는다.
    - 원본 → coalesce → **일부 파티션을 합친 상태**
        
        ** 그냥 갖다 붙이는 거라 shuffle 이 아니다. (shuffle=false 의 경우)
        

![스크린샷 2026-03-01 오후 2.27.31.png](Partitioning/%E1%84%89%E1%85%B3%E1%84%8F%E1%85%B3%E1%84%85%E1%85%B5%E1%86%AB%E1%84%89%E1%85%A3%E1%86%BA_2026-03-01_%E1%84%8B%E1%85%A9%E1%84%92%E1%85%AE_2.27.31.png)

1. **Dependencies (의존성)**
    - **Narrow Dependency (좁은 의존성)**
    : 하나의 입력(부모) 파티션이 하나의 출력(자식) 파티션에 영향을 준다.
        - ex: map, filter, flatMap, projection …
    - **Wide Dependency (넓은 의존성)**
        
        : 하나의 입력(부모) 파티션이 여러 개의 출력(자식) 파티션에 영향을 준다. → **Shuffle**
        
        - ex: grouping, join, sorting …

1. **Repartition & Coalesce**
    - **Repartition**
        - 지정한 파티션 개수를 가진 새로운 RDD 반환 ← 파티션 개수 및 크기 제어 필요
        - wide transformation ← 항상 shuffle 발생
        - 데이터의 균등 분배를 가장 우선시함
    - **Coalesce**
        - 지정한 파티션 개수를 가진 새로운 RDD 반환
        - 주로 파티션 수를 줄일 때 사용
        - narrow transformation
        - 데이터의 균등 분배는 보장되지 않음 ← skew 발생 가능성 up
        - coalesce 는 파티션 수를 줄일 때 사용하지만, 늘릴 수도 있다. 
        이 경우에는 내부적으로 repartition 과 동일하다.
            
            ![스크린샷 2026-03-01 오후 2.43.05.png](Partitioning/%E1%84%89%E1%85%B3%E1%84%8F%E1%85%B3%E1%84%85%E1%85%B5%E1%86%AB%E1%84%89%E1%85%A3%E1%86%BA_2026-03-01_%E1%84%8B%E1%85%A9%E1%84%92%E1%85%AE_2.43.05.png)
            
2. **언제 무엇을 사용할지**
    - **Repartition**
        - 병렬성을 증가시키고 싶을 때
        - 파티션 크기를 제어하고 싶을 때
        - 데이터를 균등하게 재분배하고 싶을 때
    - **Coalesce**
        - 파티션 수를 줄이고 싶을 때
        - 성능을 개선하고 싶을 때

1. **실습**
    
    ```scala
    package part4partitioning
    
    import org.apache.spark.sql.SparkSession
    
    object RepartitionCoalesce {
    
      // 1. SparkSession 생성
      // local[*] → 현재 머신의 모든 CPU 코어를 사용
      // 즉, 파티션 개수는 보통 "가상 코어 수"로 자동 설정됨
      val spark = SparkSession.builder()
        .appName("Repartition and Coalesce")
        .master("local[*]")
        .getOrCreate()
    
      val sc = spark.sparkContext
    
      // 2. 1부터 10,000,000까지 숫자 RDD 생성
      // 기본 파티션 수는 보통 "가상 코어 개수"
      val numbers = sc.parallelize(1 to 10000000)
    
      // 현재 RDD의 파티션 개수 출력
      println(numbers.partitions.length)
    
      // -------------------------------------------------------
      // 3. repartition
      // -------------------------------------------------------
      // repartition(2)
      // - 내부적으로 shuffle 발생
      // - 데이터가 모든 executor 사이에서 재분배됨
      // - 균등하게 2개 파티션으로 나눔
      // - 비용이 큼 (네트워크 이동 발생)
      val repartitionedNumbers = numbers.repartition(2)
    
      // count() ->> Action
      repartitionedNumbers.count()
    
      // -------------------------------------------------------
      // 4. coalesce (기본값 shuffle = false)
      // -------------------------------------------------------
      // coalesce(2)
      // - shuffle 없음
      // - 기존 파티션을 단순히 "붙임"
      // - 네트워크 이동 없음
      // - 비용 낮음
      // - 하지만 데이터 skew 위험 존재
      val coalescedNumbers = numbers.coalesce(2)
    
      coalescedNumbers.count()
    
      // -------------------------------------------------------
      // 5. 강제로 shuffle 시키기
      // -------------------------------------------------------
      // coalesce(2, true)
      // - shuffle = true
      // - 사실상 repartition과 동일
      // - Wide Dependency
      // - 데이터 균등 재분배
      val forcedShuffledNumbers = numbers.coalesce(2, true)
    
      // 아직 action이 없으므로 실행은 안 됨
      // 필요하면 count() 추가 가능
    
      // -------------------------------------------------------
      // 6. 프로그램 유지
      // -------------------------------------------------------
      // Spark UI (localhost:4040) 확인을 위해 대기
      // DAG / Stage / Shuffle 확인 가능
      def main(args: Array[String]): Unit = {
        Thread.sleep(10000000)
      }
    }
    ```

# **2. Partitioning Problems**

- **파티셔닝이 왜 중요한가**
    
    Spark 에서 모든 병렬성은 파티션에서 나온다.
    
    - 파티션 = 병렬 실행 단위
    - 파티션 개수 = 동시에 실행 가능한 Task 개수
    - 파티션 불균형 → 성능 저하

- **권장 사양**
    - 파티션당 10MB ~ 100MB
    - 코어 수 × 2~4배
    
    ```scala
    package part4partitioning
    
    import org.apache.spark.sql.SparkSession
    
    object PartitioningProblems {
      val spark = SparkSession.builder()
        .appName("Partitioning Problems")
        .master("local[*]")
        .getOrCreate()
    
      // ----------------------------------------------
      // 파티션 실험 함수
      // nPartitions: 사용할 파티션 개수
      // ----------------------------------------------
      def processNumbers(nPartitions: Int) = {
        val numbers = spark.range(100000000) // (약 800MB)
        
        // repartition은 shuffle 발생 (Wide Dependency)
        val repartitionedNumbers = numbers.repartition(nPartitions)
    
        // 파티션 개수 확인
        println(s"Actual partitions: ${repartitionedNumbers.rdd.getNumPartitions}")
    
        // cache: 이후 연산을 빠르게 하기 위해 메모리에 저장
        repartitionedNumbers.cache()
    
        // count()는 Action → 여기서 실제 실행 발생
        // repartition + shuffle 실행됨
        repartitionedNumbers.count()
        repartitionedNumbers.selectExpr("sum(id)").show()
        repartitionedNumbers.unpersist()
      }
    
      def main(args: Array[String]): Unit = {
    
        // 2개 → 너무 적은 파티션 (Under-partitioning)
        processNumbers(2)
    
        // 적절한 구간
        processNumbers(8)
        processNumbers(16)
    
        // 점점 많아지는 구간
        processNumbers(50)
    
        // 위험... 터질 뻔
        // processNumbers(200)
        // processNumbers(2000)
        // processNumbers(20000)
    
        println("Experiment finished safely")
        Thread.sleep(60000)
      }
    }
    ```
    

1. **문제 상황**
    - **파티션이 너무 적은 상황 (Under-partitioning)**
        
        → 병렬성 ↓, 처리 속도 ↓, 자원 낭비
        
    - **파티션이 너무 많은 상황 (Over-partitioning)**
        
        → Task 생성 오버헤드 증가, 스케줄링 비용 증가, 작은 파일 수천 개 생성
        
    - **데이터 스큐 (Data Skew)**
        
        : 특히 key 기반 연산에서 특정 파티션에 데이터가 몰리는 현상이 발생함
        
    - **잘못된 파티션 전략**
        
        : Spark 의 기본 파티션 전략은 hash partitioning 이 적용된다.
        
        **** 왜 기본이 range 가 아니고 hash 인가**
        
        : 해시는 우선 빠르고, 일반적으로는 균등하게 동작한다. 구현이 간단하고 대부분 데이터는 극단적이지 않기 때문..
        만약에 해시를 사용하고서 데이터 스큐가 발생하면 그건 키 쏠림 문제
        그럴 때는 salting 으로 키를 인위적으로 분산시킨다.
        
        ```scala
        hash(key) % numPartitions
        ```
        
        → key 분포가 균등하지 않으면, 파티션도 균등하지 않다.
        
    - **파일을 저장하는 상황**
        
        : 파티션 200개면 데이터 파일 200개 생성
        
        → namenode 부하 증가, 작은 파일 문제 
        
    - **repartition vs coalesce 오용 문제**
        - repartition → shuffle 비용 폭발
        - coalesce → 데이터 균등 분배 안해서 데이터 스큐 위험 증가

1. **해결 방법**
    - **SizeEstimator 방식**
        
        ```scala
          // 1 - use size estimator
          def dfSizeEstimator() = {
            val numbers = spark.range(100000)
            **println(SizeEstimator.estimate(numbers)) // usually works, not super accurate, within an order of magnitude - larger number**
            // measures the memory footprint of the actual JVM object backing the dataset
            numbers.cache()
            numbers.count()
          }
        ```
        
        **JVM 의 객체 크기를 재는 방법**으로, 메모리 튜닝용에 가깝다.
        
        즉 Spark 내부에 객체가 메모리를 얼마나 차지하는지 (실제 물리 데이터 크기랑 다를 가능성 큼 ← 과장해서 측정 ← 객체 메타데이터+레퍼런스+내부 구조까지 포함)
        
    - **Query Plan 기반 추정**
        
        ```scala
          // 2 - use query plan
          def estimateWithQueryPlan() = {
            val numbers = spark.range(100000)
            println(numbers.queryExecution.optimizedPlan.stats.sizeInBytes) // accurate size in bytes for the DATA
          }
        ```
        
        **실제 데이터 크기를 재는 방법**으로, **Catalyst** 가 계산한 데이터 크기이다.
        
        비교적 정확하고, 실무에서 가장 많이 쓰인다고 한다.
        

1. **실무에서 자주 발생하는 문제와 해결 방법**
    - 조인 후 한쪽 태스크만 오랜 시간이 걸리는 경우 → key skew
    - CPU 사용률 낮음 → 파티션 수 부족
    - Job 느림+Stage 많음 → 불필요한 shuffle

# **3. Shuffle Partitioning**

1. **shuffle**
    
    : spark 에서 가장 무거운 연산으로, Executor 간에 데이터를 재분배하는 과정이다. 즉, 한 노드에 있던 데이터를 다른 노드로 네트워크를 통해 이동시키는 과정이다.
    
    - 같은 key 를 같은 파티션으로 모아야하는 연산, 특히 Wide transformation 에서 발생한다.
        
        → 디스크 I/O, 네트워크 I/O, 직렬화/역직렬화, 메모리 사용 증가 … 
        

1. **shuffle partition** 
    
    : 셔플이 발생하면 spark 는 데이터를 몇 개의 파티션으로 나눌 것인지에 대해 결정한다. 그리고 이 개수를 shuffle partition 이라고 한다.
    
    ```scala
    spark.sql.shuffle.partitions=200 // 기본값 200
    ```
    
    - 셔플이 발생했을 때 기본적으로 생성되는 200개의 파티션은 모든 환경에 맞는 값이 아니다.
        - **너무 적을 경우**
            - 파티션당 데이터 과다, GC 증가, OOM 가능성, Task 오래 걸림 ⇒ 병렬성 부족
        - **너무 많을 경우**
            - Task 수 증가, 스케줄링 오버헤드 증가, 작은 파일 문제, CPU 낭비 ⇒ 관리 비용 증가
        - **최적의 계산법**
            - **권장 기준** : Partition 당 10-100MB (압축 해제 기준)
            
            → shuffle partitions ≈ 100
            
        - **Adaptive Query Execution (AQE)**
            
            : 최신 spark 에서는 자동으로 shuffle partition 수를 조정해준다. 
            
            ```scala
            spark.conf.set("spark.sql.adaptive.enabled", true)
            ```
            
    - SQL/DF 의 경우 조정을 할 수 있다.
        
        ```scala
        spark.conf.set("spark.sql.shuffle.partitions", 50)
        // RDD 에는 적용되지 않음
        ```
        

1. **shuffle 이 무거운 이유**
    
    shuffle 는 3단계를 진행한다.
    
    - **Map 단계** : 각 executor 가 데이터를 정렬하고 파일로 쓴다.
    - **Shuffle write** : 디스크에 임시 파일을 생성한다.
    - **Shuffle read** : 다른 executor 가 파일을 읽어간다.
    
    ⇒ 메모리 → 디스크 → 네트워크 → 메모리

# **4. Custom Partitioners**

1. **Partitioners** 
: key-value RDD 에서 어떤 key 를 어떤 파티션에 넣을지 결정하는 전략으로 ((key) → partition 번호), RDD 에서만 의미가 있고, DF 는 내부적으로 처리된다.
    - **RDD 내부에 포함된 정보** : 데이터, dependency, partitions, partitioner (Option[Partitioner])
        
        → 이때 partitioner 는 해당 RDD 가 key 기반 연산을 수행하고자 할 때 생성이 된다.
        
    
2. **Partitioners 가 중요한 이유**
    
    : spark 는 같은 파티셔너+같은 파티션 수 일 경우 → 같은 키는 반드시 같은 파티션 번호로 간다고 생각한다.
    
    ```scala
    partition = hash(key) % 4
    ```
    
    만약에 이 규칙이 두 개의 RDD 에 동일하게 적용되어 있다면, 
    
    ```scala
    // ex: key = “A”
    hash("A") % 4 = 2 ->> A key 는 항상 2번 파티션에 위치한다.
    ```
    
    그러면 RDD1 의 A 도 partition 2, RDD2 의 A 도 partition 2 에 위치한다. → 이미 같은 위치에 있다고 판단해서 셔플이 발생하지 않는다. 
    ⇒ shuffle 을 하는 기준은 executor 가 아니라, key 가 같은 partiton 번호에 있느냐가 판단 기준이다.
    **그래서 partitioners 는 shuffle 을 줄이는 열쇠이기 때문에, mapValues/filter 와 같이 키값이 변하지 않는 연산을 통해서 파티셔너를 유지하면 좋다.**
    

1. **Spark 기본 파티셔너 종류**
    - **HashPartitioner (기본값)**
        
        ```scala
        partition = hash(key) % numPartitions
        ```
        
        → 빠르고, 데이터가 이상하지 않는 이상 균등하게 분배되지만, 아니라면 스큐 발생 가능성이 있다.
        
    - **RangePartitioner : 정렬 기반 분산**
        
        ```scala
        partition = hash(key) % numPartitions
        ```
        
        → 정렬 연산에 유리하고, 범위 기반 쿼리 처리에 유리하지만 구현 비용이 있다.
        
    - **CustomPartitioner**
    
2. **RDD 와 DF 와의 차이**
    
    : DF 는 파티셔닝 로직을 직접 제어할 수는 없지만, 특정 규칙을 따른다.
    
    - sort/orderBy ⇒ RangePartitioning
        
        → **sort 또는 orderBy를 사용하면 RangePartitioning이 적용된다**
        
    - aggregation by key ⇒ HashPartitioning
        
        → **key 기준으로 집계(aggregation)를 하면 HashPartitioning이 적용된다**
        
    - join ⇒ both DFs obey HashPartitioning
        
        → **join을 하면 두 DataFrame 모두 HashPartitioning을 따른다**
        
    - repartition with a number ⇒ RoundRobinPartitioning
        
        → **숫자만 지정해서 repartition 하면 RoundRobinPartitioning이 적용된다**
        
    - repartition by column ⇒ HashPartitioning
        
        → **컬럼 기준으로 repartition 하면 HashPartitioning이 적용된다**
        
    
3. **실습**
    
    ```scala
    package part4partitioning
    
    import org.apache.spark.{HashPartitioner, Partitioner, RangePartitioner}
    import org.apache.spark.sql.SparkSession
    import scala.util.Random
    
    object Partitioners {
    
      // SparkSession 생성
      val spark = SparkSession.builder()
        .appName("Partitioners")
        .master("local")
        .getOrCreate()
    
      val sc = spark.sparkContext
      val numbers = sc.parallelize(1 to 10000)
    
      // 일반 RDD는 partitioner 정보가 없다 (None 출력)
      println(numbers.partitioner) // None
    
      // 파티션 개수를 3개로 재분배 (RoundRobin 방식)
      // 단순 데이터 재배치이므로 partitioner 정보는 여전히 없음
      val numbers3 = numbers.repartition(3)
      println(numbers3.partitioner) // None
    
      // -----------------------------------------
      // Key-Value RDD는 partitioning 제어 가능
      // -----------------------------------------
    
      // (key, value) 형태로 변환
      // key = n % 10 (0~9)
      val keyedNumbers = numbers.map(n => (n % 10, n)) // RDD[(Int, Int)]
    
      // HashPartitioner 적용 (4개 파티션)
      val hashedNumbers = keyedNumbers.partitionBy(new HashPartitioner(4))
    
      /*
        동작 원리:
        partition = hash(key) % 4
    
        같은 key는 반드시 같은 partition에 저장됨
    
        아래 연산들의 전제 조건:
          - combineByKey
          - groupByKey
          - aggregateByKey
          - foldByKey
          - reduceByKey
    
        또한 join 시,
        두 RDD 모두 partitioner가 없을 경우 shuffle이 발생하는데
        HashPartitioner가 있으면 shuffle 최적화 가능
       */
    
      // RangePartitioner 적용 (5개 파티션)
      val rangedNumbers = keyedNumbers.partitionBy(new RangePartitioner(5, keyedNumbers))
    
      /*
        동작 원리:
        key의 범위를 기준으로 partition 배정
    
        예:
        key 범위 0-9라면,
        작은 값부터 순서대로 5개 구간으로 나눔
    
        예시 (0~1000 범위라고 가정하면):
        Int.MinValue ~ 200     → partition 0
        200 ~ 400              → partition 1
        400 ~ 600              → partition 2
        600 ~ 800              → partition 3
        800 ~ Int.MaxValue     → partition 4
    
        특징:
        - key 범위 기준으로 정렬된 구조
        - sortByKey의 전제 조건
       */
    
      // 이미 RangePartitioner가 적용되어 있으므로
      // sortByKey 실행 시 shuffle이 발생하지 않음
      rangedNumbers.sortByKey()
    
      // -----------------------------------------
      // Custom Partitioner 실습
      // -----------------------------------------
    
      // 랜덤 문자열 생성 함수
      def generateRandomWords(nWords: Int, maxLength: Int) = {
        val r = new Random()
        (1 to nWords).map(_ =>
          r.alphanumeric
            .take(r.nextInt(maxLength)) // 길이 랜덤
            .mkString("")
        )
      }
    
      // 랜덤 단어 1000개 생성
      val randomWordsRDD = sc.parallelize(generateRandomWords(1000, 100))
    
      /*
        목표: 단어 길이 기준으로 partition 나누기
        → 같은 길이의 단어는 같은 partition에 위치하게 만들기
       */
    
      // 각 단어에 대해 'z' 개수 세기
      // (word, z의 개수)
      val zWordsRDD = randomWordsRDD.map(word =>
        (word, word.count(_ == 'z'))
      ) // RDD[(String, Int)]
    
      // Custom Partitioner 정의
      class ByLengthPartitioner(override val numPartitions: Int)
        extends Partitioner {
    
        // key가 어느 partition에 갈지 정의
        override def getPartition(key: Any): Int = {
    
          // key는 word (String)
          // 단어 길이를 numPartitions로 나눈 나머지로 partition 결정
          key.toString.length % numPartitions
        }
      }
    
      // Custom Partitioner 적용 (100개 파티션)
      val byLengthZWords =
        zWordsRDD.partitionBy(new ByLengthPartitioner(100))
    
      /*
        결과: 단어 길이가 같은 단어들은
        length % 100 값이 같으므로
        같은 partition에 배치됨
       */
    
      // 메인 함수
      def main(args: Array[String]): Unit = {
    
        // 각 partition 별로 데이터 출력
        byLengthZWords.foreachPartition { partition =>
          partition.foreach(println)
        }
      }
    }
    ```
